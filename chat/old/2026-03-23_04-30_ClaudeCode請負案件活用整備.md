# Claude Code 請負案件活用整備 — 課題と対策案

> 作成日：2026-03-23
> 目的：社内の請負案件でClaude Codeを活用するための整備観点の整理

---

## 整理した観点一覧

ご指摘の2点に加え、請負案件特有の観点を整理した。

| # | 観点 | 重要度 | 理由 |
| - | ---- | ------ | ---- |
| 1 | 料金体系 | ★★★ | プロジェクト収益に直結 |
| 2 | セキュリティ | ★★★ | 顧客情報・機密コードの漏洩リスク |
| 3 | 法的・契約上のリスク | ★★★ | 利用規約違反・顧客との契約との齟齬 |
| 4 | 成果物の品質と責任 | ★★☆ | AIが生成したコードの不具合責任の所在 |
| 5 | 運用・ガバナンス | ★★☆ | 統制なく使われると統一できない |
| 6 | 知的財産権 | ★★☆ | 生成コードの帰属と顧客への開示義務 |

## 1. 料金体系

### 課題

| 課題 | 詳細 |
| ---- | ---- |
| **コスト予見が困難** | トークン従量課金のため、案件規模・エンジニア数によってコストが変動し、見積り精度が下がる |
| **コストの案件別帰属が難しい** | 複数案件を並行している場合、どの案件にいくらのコストがかかったか追跡できない |
| **上振れリスクの管理** | コード補完・長いコンテキストを使うと1セッションで大量トークンを消費する可能性がある |
| **ライセンスモデルの選択** | Claude Code は Anthropic API の従量課金と Claude Pro/Max の定額課金があり、どちらが有利か案件規模次第 |

### 対策案

- **コスト試算基準の策定**：過去の試用データを元に「エンジニア×日単価」の目安コストを算出し、見積りテンプレートに組み込む
- **案件別API Keyの発行**：プロジェクトごとにAPI Keyを分け、コスト配賦を自動化する
- **使用量アラートの設定**：Anthropic Consoleのコスト通知を案件単位の閾値で設定する
- **定額プランとの比較**：小規模案件は Claude Max（定額）、大規模チームはAPI従量課金で使い分けるルールを作る
- **見積り上の加算方針**：原価として案件見積りに明示的に計上する（例：開発費の○%をAI利用料として積算）

## 2. セキュリティ（顧客コード外部送信問題の深掘り）

### 問題の構造と前提の再確認

Claude Code を請負開発で使う場合、顧客コードが必然的に Anthropic のサーバーへ送信される。
**大多数の顧客企業は「外部クラウドサービスへの機密情報アップロード禁止」というポリシーを持っており、これは例外ではなく標準。**
「合意を取れれば使える」という発想では大半の案件で使えない。

本質的な問いは **「Anthropic への送信を回避しながら AI 支援を得る手段があるか」** であり、その選択肢を評価する。

---

### 回避策の選択肢と評価

#### 選択肢①：Amazon Bedrock 経由で Claude Code を使う（最有望）

Claude Code は Anthropic への直接接続に加え、**Amazon Bedrock をバックエンドとして使う設定が可能**。

```bash
# Claude Code を Bedrock モードで起動
claude --bedrock --region ap-northeast-1
```

**データの流れが変わる：**

```text
通常モード：開発PC → Anthropic サーバー（米国）
Bedrock モード：開発PC → 顧客/自社の AWS アカウント内 → Bedrock API
```

| 評価軸 | 内容 |
| ------ | ---- |
| **顧客ポリシーとの整合** | Anthropic ではなく AWS にデータが送られるため、「AWS 信頼済み」の顧客には受け入れられやすい |
| **既存 AWS 契約の流用** | 顧客が既に AWS と DPA・BAA を締結している場合、追加契約不要で利用できる |
| **データ保存** | Bedrock はデフォルトでリクエスト/レスポンスをログに残さない。顧客の AWS アカウント内で完結 |
| **モデル品質** | Anthropic の Claude モデルと同一（Sonnet / Opus 等） |
| **コスト** | Bedrock の従量課金（Anthropic API と大差なし） |
| **制約** | 利用可能なリージョンとモデルのバージョンが Anthropic 直販より遅れる場合がある |

**顧客への説明のポイント：**
「コードは貴社（または当社）の AWS アカウント内でのみ処理され、Anthropic のサーバーには送られません」

#### 選択肢②：ローカルLLM（コードが一切外に出ない）

開発マシン上で LLM を動かし、Claude Code の代わりに使う。
Claude Code 自体はローカル LLM に対応していないため、**別の AI コーディング支援ツールと組み合わせる**形になる。

| ツール | 用途 | 品質感 |
| ------ | ---- | ------ |
| Ollama + Qwen2.5-Coder 32B | コード補完・説明 | Claude Sonnet の 6〜7 割程度 |
| Ollama + DeepSeek-Coder-V2 | コード生成 | 同上 |
| Continue（VS Code 拡張） | ローカル LLM をコーディング支援に統合 | ツール依存 |

**制約：**

- GPU（VRAM 16GB 以上推奨）が必要。CPU のみだと実用的な速度が出ない
- Claude Code の「ターミナル操作・ファイル操作まで自律的に行う」機能は使えない
- 高品質なモデルほどハードウェア要件が高い

**向いている状況：** 顧客環境内に開発用サーバーを置ける案件、セキュリティ要件が最上位の官公庁・金融案件

#### 選択肢③：コードのサニタイズ（運用でカバー）

顧客コードから機密性の高い部分を除去・抽象化してから Claude Code に投げる運用ルールを設ける。

**サニタイズの方法：**

| 対象 | 処理 |
| ---- | ---- |
| 認証情報・APIキー | `.claudeignore` で物理的に除外 |
| 顧客固有の業務ロジック名 | 汎用的な名前に置き換えてから質問（例：`invoiceCalc()` → `calcAmount()`） |
| 個人情報を含むデータ | ダミーデータに差し替えてからコンテキストに渡す |
| 機密性の高いアルゴリズム | 「このような処理をしたい」と文章で説明し、コードは渡さない |

**限界：**
コンテキストが減ると回答精度が落ちる。複雑なリファクタリングや既存コードへの機能追加では効果が半減する。

**向いている状況：** 質問の対象が部分的・局所的な案件（新規モジュール開発、アルゴリズム相談等）

#### 選択肢④：AI 活用範囲を実装コード以外に限定

顧客コードを送らずに AI を活用できる領域に絞る。

| 活用領域 | 内容 | 顧客コード送信 |
| -------- | ---- | -------------- |
| 設計・アーキテクチャ相談 | 「こういう構成にしたいがどう設計するか」 | なし |
| ドキュメント生成 | 仕様書・API ドキュメントの下書き | 最小限 |
| テストコードのひな形生成 | インターフェース定義のみ渡す | 最小限 |
| ボイラープレート生成 | 新規ファイルのスケルトン作成 | なし |
| 技術調査・比較 | ライブラリ選定・技術トレンド調査 | なし |

**限界：** 既存コードの修正・バグ調査には向かない。生産性向上の恩恵が最も少ない使い方。

---

### 選択肢の比較まとめ

| 選択肢 | Anthropic 送信 | 品質 | 導入コスト | 顧客ポリシー通過しやすさ |
| ------ | -------------- | ---- | ---------- | ------------------------ |
| **① Bedrock 経由** | なし（AWS止まり） | Claude と同等 | 低〜中 | ★★★（AWS 信頼済みなら） |
| **② ローカルLLM** | なし | Claude の 6〜7 割 | 高（GPU必要） | ★★★（完全閉域） |
| **③ サニタイズ運用** | あり（機密除去済み） | 制限あり | 低（運用負荷あり） | ★★（グレーゾーン） |
| **④ 活用範囲限定** | 最小限 | 制限あり | 低 | ★★★（ただし効果も限定） |

---

### 現実的な推奨方針

```text
受注前の顧客ポリシー確認
  ├── AWS を信頼済みの顧客 → ① Bedrock 経由で Claude Code を使う
  ├── 完全閉域が必要な顧客 → ② ローカルLLM（GPU環境の整備が前提）
  ├── 条件交渉の余地がある顧客 → ③ サニタイズ運用 + 契約上の合意
  └── AI 利用そのものが NG の顧客 → ④ 活用範囲限定 または AI 不使用で受注
```

**実態として：**
Bedrock 経由が「品質を落とさずにポリシーをクリアできる」唯一の現実的な選択肢。
ローカルLLM は品質・導入コストの壁があり、今すぐ全案件に展開できるものではない。

---

### 残る論点：Bedrock でも顧客が納得しないケース

顧客が「AWS も外部サービスだから NG」というポリシーの場合は手段がなくなる。
その場合の現実的な対応：

1. **顧客 AWS アカウント内に Bedrock 環境を構築してもらう** → 顧客のアカウント内で完結するため「外部送信」に該当しない解釈が成立する
2. **ローカルLLM のみ使用** → GPU 環境のコストを見積りに乗せる
3. **AI 不使用で受注し、単価・工数を見直す** → AI 活用前提の見積りを出さない

## 3. 法的・契約上のリスク

### 課題

| 課題 | 詳細 |
| ---- | ---- |
| **利用規約の商用利用条件** | Anthropic利用規約の「禁止用途」が業種・案件内容に抵触しないか確認が必要 |
| **顧客との契約との齟齬** | 顧客側の「外部サービスへのデータ送信禁止」条項や秘密保持条項と抵触する場合がある |
| **再委託・再利用の制約** | 成果物に含まれるAI生成コードの顧客への権利移転が利用規約上問題ないか |
| **コンプライアンス規制** | 金融・医療・政府系案件では特定の規制（FISC・医療情報安全管理ガイドライン等）への準拠が必要 |

### 対策案

- **法務レビューの実施**：Anthropic利用規約と顧客契約の双方を法務が確認する（特に秘密保持・データ処理・再委託条項）
- **AI利用開示条項の標準化**：請負契約のひな形に「AI開発支援ツール利用条項」を追加し、顧客の同意を取得するプロセスを標準化する
- **業種別のNG案件リスト作成**：金融・医療・官公庁など高機密性が求められる業種は、利用前に個別審査するフローを設ける
- **成果物の権利帰属の明確化**：「AI生成部分を含む成果物の著作権は顧客に帰属する」旨を契約書に明記する

## 4. 成果物の品質と責任

### 課題

| 課題 | 詳細 |
| ---- | ---- |
| **AI生成コードの品質保証** | LLMの出力はハルシネーション・非推奨API使用・セキュリティホールを含む場合がある |
| **レビュー工数の過小見積り** | 「AIが書いたから早い」という期待から、レビュー工数を削減すると品質が低下する |
| **不具合時の責任範囲** | AIが生成したコードに起因する不具合について、誰がどこまで責任を負うかが曖昧になりやすい |
| **テスト戦略への影響** | AI生成コードをレビューなしで使うと、テストが「AIの思い込みを確認するだけ」になるリスク |

### 対策案

- **AI利用を前提とした品質基準の再設定**：コードレビュー必須・静的解析（ESLint / Snyk等）必須・テストカバレッジ基準を維持する
- **「AI生成コードの使用割合」の記録**：ファイル・モジュール単位でAI利用の有無を記録し、レビュー優先度を決める
- **レビュー工数の見積りルール化**：AI活用による工数削減分の一部をレビュー工数に再配分するガイドラインを設ける
- **責任範囲の契約上の明記**：最終納品前に人間のレビューを経ることを契約に記載し、不具合責任は受託者が通常通り負う旨を明確にする

## 5. 運用・ガバナンス

### 課題

| 課題 | 詳細 |
| ---- | ---- |
| **使い方の個人差が大きい** | 統一ルールがないとプロジェクト間で品質・コストに大きなばらつきが出る |
| **利用ログ・監査証跡の欠如** | 誰が何を入力したか追跡できないと、インシデント時の原因調査ができない |
| **メンバーのリテラシー格差** | 使い方を知らないメンバーが多いと、効果が出ないまたは誤用が起きる |
| **ツールの更新追跡** | Claude Codeの機能・APIの仕様変更が頻繁で、社内ルールが陳腐化する |

### 対策案

- **社内利用ガイドラインの策定**：使って良いこと・禁止事項・コンテキストに含めてよいものを1ページにまとめ全員に周知する
- **CLAUDE.mdによるプロジェクト設定の統一**：各リポジトリに `CLAUDE.md` を配置し、プロジェクト固有のルール・禁止事項・コーディング規約を共有する
- **利用ログの取得**：Anthropic API経由で利用する場合、リクエスト/レスポンスログを社内ログ基盤に保存する仕組みを設ける
- **初期研修・定期アップデート勉強会**：利用開始前の研修と、四半期ごとのアップデート共有会をセットにする
- **ガイドラインのオーナーを設定**：ツールの変化を追跡し、ガイドラインを更新する担当者（または委員会）を決める

## 6. 知的財産権

### 課題

| 課題 | 詳細 |
| ---- | ---- |
| **AI生成コードの著作権帰属** | 日本著作権法上、AI単独で生成したコードには著作権が発生しないとされているが、解釈が固まっていない |
| **第三者著作物の混入** | 学習データ由来のコードが出力に含まれ、OSS・他社コードと類似するリスク |
| **顧客への開示義務** | 成果物にAI生成コードが含まれることを顧客に開示すべきか、顧客が気づかずに使用するリスク |

### 対策案

- **AI利用開示ポリシーの策定**：成果物へのAI活用を顧客に開示する基準を設ける（「AIを補助的に使用している」旨を納品書に記載するなど）
- **類似コードの検出**：GitHub Copilot Filterに相当する類似検出の仕組みを導入し、著名OSS由来コードの混入をチェックする
- **法務・社外弁護士との連携**：AI成果物の権利帰属に関する法解釈を定期的に確認し、契約ひな形に反映する

## まとめ：優先度別の整備ロードマップ案

```text
Phase 1（即時）
  ├── セキュリティガイドライン策定（送信禁止データの定義）
  ├── 顧客向けAI利用開示条項の契約ひな形への追加
  └── プロジェクト別APIキー管理の仕組み構築

Phase 2（1〜3ヶ月）
  ├── コスト積算・見積りルールの整備
  ├── CLAUDE.md テンプレートの標準化
  ├── 社内研修の実施
  └── 業種別NG案件リストの作成

Phase 3（3〜6ヶ月）
  ├── 利用ログ・監査証跡の仕組み構築
  ├── AI成果物の品質基準・レビュールールの定着
  └── Enterprise Planへの移行検討（必要に応じて）
```
