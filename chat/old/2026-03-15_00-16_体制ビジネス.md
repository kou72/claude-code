# 体制ビジネス：構造整理

体制ビジネスの構造を3つの切り口で整理する。

---

## パターンA：登場人物マップ

会社単位と人物単位の2層で整理する。

![パターンA：登場人物マップ](charts/fig1_players.png)

### 会社単位（4社）

| 会社 | 役割 |
| --- | --- |
| エンドユーザー | システムを発注する側。予算・要件・決裁権を持つ |
| プライマリーベンダー | 一次請け。エンドユーザーと直接契約し、全体を管理する |
| 体制企業 | PLの自社。体制まるごとでプライマリーベンダーに入る |
| 要員企業 | SEを供給する会社。体制企業の下に入る |

### 人物単位

| 人物 | 所属 | 役割 |
| --- | --- | --- |
| クライアント（担当者） | エンドユーザー | 要件定義・評価・意思決定 |
| PM（マネージャー） | プライマリーベンダー | 体制依頼・品質管理・クライアント対応 |
| PL（体制リーダー） | 体制企業 | 現場指揮・SE管理・プライマリーベンダー対応 |
| SE（エンジニア） | 要員企業 | 実装・開発・PLの指示を受けて稼働 |
| 各社の営業 | 各社 | 下流→上流への提案・関係維持 |

実線（上→下）は発注・業務の流れ。破線（下→上）は各社営業による提案の流れ。

## パターンC：お金の流れ／情報の流れ（分離）

「お金」と「情報」を分けて見ると、構造の非対称性が見えてくる。

### C-1：お金の流れ

![パターンC-1：お金の流れ](charts/fig3_money.png)

体制単価は一括で受け取り、PL稼働分とSE分に分配される。
SE1人増えるごとにマージンが増えるスケーラブル構造。

### C-2：情報の流れ（フェーズ別）

![パターンC-2：情報の流れ](charts/fig4_info.png)

フェーズごとに情報の主役が変わる。
完了フェーズの評価フィードバックはPMからPLへ届くが、その先（体制企業・SE・要員企業）には伝わらない。

## パターンD：お金の流れ（4社全体）

エンドユーザーが払う全体単価が、4社を経由してどう分配されるか。

![パターンD：お金の流れ（4社全体）](charts/fig5_money_full.png)

3層（A・B・C）でマージンが抜かれる構造。SEに届く報酬は全体単価の一部に過ぎない。
体制企業のマージンBは「PLへの支払い + SEへの支払いの差額」であり、SE人数が増えるほど大きくなる。

## パターンE：契約構造

誰と誰が契約しているか。お金・情報の流れとは別に見る必要がある。

![パターンE：契約構造](charts/fig6_contracts.png)

エンドユーザーはシステム開発全体を一括請負でプライマリーベンダーに発注する。
プライマリーベンダーより下流はSES参画（準委任）に切り替わり、成果物ではなく稼働ベースの契約になる。
この境界がリスク・責任の所在を分ける構造的な切れ目になっている。

---

## パターンF：意思決定構造

誰が何を決めるか。各意思決定の権限と依存関係を整理する。

![パターンF：意思決定構造](charts/fig7_decisions.png)

SE選定の最終権限はPLにあるが、候補提案は要員企業営業が担う。
体制承認はPMが行い、クライアントへの報告はPMを通じて行われる。
評価フィードバックはPM→PLには届くが、要員企業・SEへは伝わらない構造になっている。

## Step 2：オペレーション時系列

各フェーズで誰が何をするか・何を渡すかを時系列で追う。

![Step 2：オペレーション時系列](charts/fig8_ops_timeline.png)

### 各フェーズのポイント

#### ① 案件受付・仮合意

体制企業営業が案件情報を入手し、PLが参加可否と収益性を判断する。
この段階では単価を仮合意するにとどまり、契約はまだ締結しない。

#### ② 体制組成 → 受注確定

PLがロール・スキル要件を定義し、要員企業営業を通じてSEを確保する。
体制案をPMに提出し承認されて初めて契約締結（受注確定）となる。
「良い体制が提案できたら受注が確定する」という条件付き構造になっている。

#### ③ 稼働中

SEの勤怠・実績がPLに集まり、PLがPMへ月次報告する。
請求処理は体制企業が担うが、根拠データ（勤怠・実績）の管理方法は属人的になりやすい。

#### ④ 完了・次へ

PMからPLへ評価フィードバックが届く。
PL・体制企業営業が次案件への種まきを行い、ループが回る。

## Step 3：課題分類

Steps 1・2で見えた事実を「課題」として4カテゴリに整理する。

![Step 3：課題分類](charts/fig9_issues.png)

| カテゴリ | 課題 | 発生フェーズ | 影響プレイヤー |
| --- | --- | --- | --- |
| 調達・プロセス効率 | SE探しが人脈・属人依存 | ②体制組成 | PL・体制企業 |
| 調達・プロセス効率 | 体制組成〜受注確定のリードタイムが長い | ①〜② | PL・体制企業・要員企業 |
| 情報の非対称性 | エンドユーザーが多重商流を契約上把握できない | 全体 | エンドユーザー |
| 情報の非対称性 | 単価・マージン構造がエンドユーザーに不透明 | 全体 | エンドユーザー |
| プロセスリスク | 受注確定前にSEを確保する空振りリスク | ①〜② | 体制企業・要員企業 |
| プロセスリスク | SE途中離脱のリスクを誰が吸収するか不明確 | ③稼働中 | 体制企業・要員企業 |
| 管理の属人性 | 勤怠・実績管理がPL個人に依存 | ③稼働中 | PL・体制企業 |
| 管理の属人性 | SE選定の判断根拠が記録されず次に活かせない | ②体制組成 | PL |

## 各パターンから見える問い

| パターン | 問い |
| --- | --- |
| A（登場人物） | 営業担当は自社内か外部エージェントか。リーダーと自社の契約形態は何か |
| B（フェーズ） | 体制組成フェーズをどう短縮するか。完了フェーズの実績をどう次につなぐか |
| C（流れの分離） | 完了後の評価情報はどこに蓄積されるか。お金の分配はどの単位で可視化されているか |
| D（4社マージン） | 各社のマージン率はどこで決まるか。エンドユーザーは単価内訳を知っているか |
| E（契約構造） | 多重下請けをエンドユーザーが把握できない構造をどう変えるか |
| F（意思決定） | SE選定・評価の情報がSE本人・要員企業に届かない構造をどう変えるか |
