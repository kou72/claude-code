# BtoB 乗り換え事例分析：SES業界マッチングプラットフォームへの設計示唆

## 概要

個人のツール乗り換えと異なり、企業（法人）の乗り換えは「組織としての意思決定」を経る。
本レポートでは、BtoBの乗り換え事例8例を分析し、「企業導入の決断メカニズム」を解明したうえで、
SES業界マッチングプラットフォームへの設計示唆を導く。

---

## 第1部：BtoB乗り換え事例8例

### 事例1：紙・FAX → EDI（電子データ交換）

**製造業・サプライチェーン管理**

#### 何から何への乗り換えか

FAXや紙の発注書・納品書・請求書のやり取りから、企業間でシステムが自動的にデータをやり取りするEDI（Electronic Data Interchange）への移行。

#### 乗り換えを決断したのは誰か

**取引先（発注元の大企業）の調達部門・情報システム部門**が先行導入し、サプライヤー（中小製造業）に対して「EDI対応を取引条件にする」と通告したことで、サプライヤー側は経営者判断として対応を迫られた。決断者は取引先から要求を受けた経営者・社長クラス。

#### 決断トリガー

- **取引先からの要件化**：「EDI非対応企業とは発注を継続しない」という通告
- コスト削減の圧力：FAXの人的処理コストが可視化された
- 競合他社がすでにEDI対応を完了し、リードタイムで差がついていた

#### 乗り換えを阻んだ抵抗と克服方法

| 抵抗 | 克服方法 |
| ---- | -------- |
| 「今のFAXで困っていない」という現場の慣性 | 大手取引先からの期限付き要求で強制的に動かざるを得なくなった |
| 初期導入コストへの懸念 | 大手発注元がシステム費用の一部を補助するケースがあった |
| 「取引先が全員EDIを使うとは限らない」という不確実性 | 大手が先行することで中小にもドミノ的に普及 |

#### 乗り換え後に「やめられなくなった理由」

受発注の自動化で翌日出荷が可能になり、「FAX対応企業より1〜2日リードタイムが短い」という競争優位が生まれた。やめると取引先要件から外れるため、戻れない構造になる。

### 事例2：Excel管理 → Salesforce（CRM）

**企業営業管理**

#### 何から何への乗り換えか

各担当者がバラバラに管理していたExcelの顧客・案件管理台帳から、Salesforce Sales Cloudによる一元管理への移行。

#### 乗り換えを決断したのは誰か

**経営者または営業部門長**。現場担当者はExcelに慣れており、むしろ変化に抵抗した。
「営業の数字が経営から見えない」「担当者が辞めたら情報が消える」という経営課題の可視化が決断を促した。

#### 決断トリガー

- **営業担当者の退職による情報喪失**：引き継ぎができず商談が消滅する事故が発生
- 投資家や親会社から「データに基づく経営管理」を求められた
- 競合他社のSalesforce導入が業界内で話題になった

#### 乗り換えを阻んだ抵抗と克服方法

| 抵抗 | 克服方法 |
| ---- | -------- |
| 「Salesforceに入力する時間がない」という現場の声 | 経営者が「入力しない者は評価しない」と宣言し、運用ルールを義務付けた |
| Excel管理のほうが柔軟という意見 | Salesforceのカスタマイズ性を示し、Excel同等の柔軟性があることを実証した |
| 導入・カスタマイズコストの高さ | 宣伝広告費の削減効果（3年で49%削減）を試算として提示した |

#### 乗り換え後に「やめられなくなった理由」

顧客データ・商談履歴・売上予測が蓄積されるにつれ、SalesforceなしではPDCAが回らなくなる。
データが資産化するため、乗り換えコスト（データ移行・再設定）が年々上昇し、実質的にロックインが完成する。

### 事例3：紙・印鑑 → 電子契約（クラウドサイン・DocuSign）

**契約締結プロセス**

#### 何から何への乗り換えか

「紙に印鑑を押して郵送」する従来の契約締結プロセスから、電子契約システムを用いたデジタル完結の契約締結へ。クラウドサインは国内15万社以上、DocuSignは世界50万社以上が導入。

#### 乗り換えを決断したのは誰か

初期は**法務部門・情報システム部門**が主導。大企業では「取引先から電子契約で送られてきた」ことが実務上の起点となり、「受信はできないと困る」という現場ニーズから始まり、その後「自社からも送信したい」へと拡大した。

#### 決断トリガー

- **コロナ禍（2020年）による在宅勤務の強制**：印鑑を押すためだけに出社が必要な矛盾が顕在化
- 取引先（特に大手・外資系）が電子契約を送付し始めた
- 電子署名法・IT書面一括法の整備で法的有効性が担保された
- 年間4,000万円規模のコスト削減事例が業界内で共有された

#### 乗り換えを阻んだ抵抗と克服方法

| 抵抗 | 克服方法 |
| ---- | -------- |
| 「法的効力があるのか」という法務部の懸念 | 弁護士ドットコムが運営するクラウドサインが「法的安心感」を担保。電子署名法の適法性を解説した資料を提供 |
| 取引先が対応できないという懸念 | 相手方はメールアドレスさえあれば署名できる仕様（アカウント不要）を採用 |
| 印鑑文化・慣習への愛着 | 経営トップが「電子契約推進」を方針として宣言 |

#### 乗り換え後に「やめられなくなった理由」

契約書のデータベース化が進み、期限管理・検索・監査対応がシステムに依存する。
一度デジタル化すると、紙に戻す工数が膨大になる。

### 事例4：メール添付 → クラウドストレージ（Box・Dropbox Business）

**企業内ファイル共有**

#### 何から何への乗り換えか

「メールにファイルを添付して送受信」するファイル共有方法から、Box・Dropbox等のクラウドストレージでのリアルタイム共有・共同編集への移行。

#### 乗り換えを決断したのは誰か

**情報システム部門が主導**するケースと、**現場チームが先行導入してからIT部門が追認**するケースの両方がある。特に外資系・IT企業では前者、コンサル・クリエイティブ系では後者が多い。

#### 決断トリガー

- **情報漏洩インシデント**：大容量ファイルのメール誤送信、PCの紛失
- テレワーク普及によりファイルサーバーへのVPNアクセスが遅延・不安定化
- 取引先や顧客から「Boxのリンクで共有してください」と指定されるケースが増加

#### 乗り換えを阻んだ抵抗と克服方法

| 抵抗 | 克服方法 |
| ---- | -------- |
| 「メールで事足りている」という慣性 | 添付ファイルのバージョン管理ミスによる業務事故を根拠に切り替えを説得 |
| セキュリティ懸念（クラウドへのデータ預託） | ISO27001・SOC2認証の提示、大企業・政府機関の採用実績を示した |
| ファイル命名規則・フォルダ構造の見直し作業 | 段階的移行（新規プロジェクトから先行適用） |

#### 乗り換え後に「やめられなくなった理由」

過去の全ドキュメントがクラウド上に蓄積され、検索・監査・コンプライアンス対応がBoxなしでは成立しなくなる。

### 事例5：電話・FAX → Slack / Microsoft Teams

**企業間・社内コミュニケーション**

#### 何から何への乗り換えか

社内連絡のメール・電話中心のコミュニケーションから、チャットツール（Slack・Teams）への切り替え。特にSlack Connectでは社外企業とも同一チャンネルで協業できるため、取引先とのメール・電話も代替される。

#### 乗り換えを決断したのは誰か

**現場の技術チーム・チャンピオンが先行**するボトムアップが多い。メルカリ・電通デジタル等では部署ごとのコミュニケーション分断が経営課題となり、**CTO・CDO級が主導**して全社統一した事例もある。

#### 決断トリガー

- テレワーク化による「リアルタイムコミュニケーション」の必要性
- 部署間・プロジェクト間の情報サイロ化が経営課題として認識された
- Slack Connectで取引先から招待されたことで、相手に合わせて導入せざるを得なくなった

#### 乗り換えを阻んだ抵抗と克服方法

| 抵抗 | 克服方法 |
| ---- | -------- |
| 「メールで十分」という管理職の抵抗 | CTOが率先してチャットワークを退室しSlackに移行するという「退路を断つ」手法（トレタ社事例） |
| ツールが増えることへの懸念 | 「メールを廃止してSlackに統一」という明確な方針設定 |
| セキュリティポリシーへの抵触 | Enterprise版の採用でセキュリティ要件をクリア |

#### 乗り換え後に「やめられなくなった理由」

過去の会話ログ・ファイル・意思決定履歴がすべてSlack/Teams内に蓄積。
Slack Connectで繋がった取引先との関係がプラットフォーム依存となり、離脱すると取引先との接点も失う。

### 事例6：紙のタイムカード → クラウド勤怠（SmartHR・freee HR）

**人事・労務管理**

#### 何から何への乗り換えか

タイムカードや紙の出勤簿による勤怠管理から、スマートフォン打刻・クラウド集計のシステムへの移行。

#### 乗り換えを決断したのは誰か

**人事部門・労務担当者**が「手集計の工数削減」を目的に提案し、**経営者が法令対応の観点**から承認するパターンが多い。

#### 決断トリガー

- **2019年の働き方改革関連法（労働時間管理の義務化）**：客観的な勤怠記録の保存が法的義務となった
- 2024年施行のフリーランス新法を含む継続的な労働法改正
- 月末の手集計・給与計算に担当者が丸2〜3日を費やしていた実態の可視化

#### 乗り換えを阻んだ抵抗と克服方法

| 抵抗 | 克服方法 |
| ---- | -------- |
| 「タイムカードで問題ない」という現場の慣性 | 労働基準監督署の是正勧告・書類送検リスクを経営者に提示 |
| 初期設定・全社展開のコスト | SmartHRの30名以下無料プランで小規模から試行 |
| 従業員のスマートフォン操作への不安 | シンプルなUI設計と丁寧な導入支援 |

#### 乗り換え後に「やめられなくなった理由」

勤怠・労務・給与データが一元連携され、年末調整・社会保険手続きもペーパーレス化が進む。
データが年々蓄積し、分析・監査対応がシステムに依存するようになる。

### 事例7：属人的採用管理 → ATS（採用管理システム）

**採用管理プロセス**

#### 何から何への乗り換えか

採用担当者それぞれが個別にExcelとメールで応募者を管理するやり方から、採用管理システム（ATS）による一元管理への移行。

#### 乗り換えを決断したのは誰か

**人事部長・採用責任者**が主導。採用規模が年100名超になると、Excelとメールの管理が破綻することが多く、採用目標未達が経営問題として浮上した段階で意思決定が下りる。

#### 決断トリガー

- 採用担当者が増員されても「新卒100名採用」という目標が達成できなかった
- 応募者への連絡漏れ・選考進捗の把握ミスが多発
- 複数の採用担当者間でのデータ共有ができず、二重アプローチが発生

#### 乗り換えを阻んだ抵抗と克服方法

| 抵抗 | 克服方法 |
| ---- | -------- |
| 「今のExcelで管理できている」という既存担当者の抵抗 | 採用目標未達・連絡ミスの実績データで問題を可視化 |
| 導入費用・設定工数への懸念 | 採用目標達成時の人件費対比でROIを計算して承認を得た |
| 担当者ごとの管理スタイルが崩れる | 共通プロセスの標準化をむしろメリットとして提示 |

#### 乗り換え後に「やめられなくなった理由」

応募者データ・選考履歴・合否理由が蓄積し、採用精度の改善や採用ブランドの管理がATSなしでは成立しなくなる。

### 事例8：個別交渉・FAX → BtoBプラットフォーム（インフォマート）

**食材・食品の受発注（飲食業界）**

#### 何から何への乗り換えか

飲食店と食材卸業者の間で行われていた電話・FAXによる受発注から、インフォマート「BtoBプラットフォーム受発注」によるデジタル受発注への移行。現在の利用企業数は全国1,265,121社、年間流通金額62兆9,565億円。

#### 乗り換えを決断したのは誰か

**大手チェーンレストランの調達・購買部門**が先行導入し、「取引継続にはインフォマート対応が必要」と卸業者に通告。卸業者側は経営判断として対応を余儀なくされた。その後、卸業者が別の飲食店に「インフォマートで取引できます」と営業し、ドミノ的に普及した。

#### 決断トリガー

- **大手取引先からの要件化**（EDIと同様のパターン）
- 仕入れコストの透明化・比較による競争圧力
- FAX対応の競合より「リードタイム1〜2日短縮」という競争優位

#### 乗り換えを阻んだ抵抗と克服方法

| 抵抗 | 克服方法 |
| ---- | -------- |
| FAX文化への慣性（特に中高年の調理スタッフ） | スマートフォンから発注できるシンプルなUI設計 |
| 全取引先がデジタル対応できないという懸念 | 「まず大口取引先だけ対応し、段階的に拡大」という移行設計 |

#### 乗り換え後に「やめられなくなった理由」

「売り手企業（卸）が圧倒的に多い」というネットワーク効果が完成。
買い手（飲食店）が他社の受発注システムを検討した際に、売り手側から「インフォマートじゃないの？」と反発が起き、他サービスへの乗り換えが実質的に阻止される。

## 第2部：企業が乗り換えを決断する5つのパターン

### パターン1：コンプライアンス・規制への対応

**「やらないといけない」から始まり「やってよかった」に変わる体験**

#### どんな規制・要求が引き金になったか

- 労働時間管理の客観的記録義務化（2019年働き方改革）→ クラウド勤怠
- 電子署名法・IT書面一括法の整備 → 電子契約
- フリーランス新法（2024年施行）→ 契約内容の明確化・記録化
- 個人情報保護法改正 → データ管理システムへの移行

#### 「やらないといけない」から「やってよかった」への転換

このパターンの本質は「外部強制によって踏み出した第一歩が、内部効率化の気づきを生む」ことにある。

法令対応として勤怠システムを入れた企業が「残業時間の分析ができるようになった」「労務トラブルの証拠が残せるようになった」と気づき、追加機能を積極採用するようになる。コンプライアンス対応が入口となり、デジタル化の恩恵を体験することで、自発的な活用に変わる。

#### 設計示唆

規制対応は「強制力があるため抵抗が低い」という最大の強みを持つ。ただし「規制が強化されなければ動かない」という受動性も持つ。先行して「規制に対応できるプラットフォーム」として位置づけることで、規制強化のたびに需要が発生する設計にできる。

---

### パターン2：経営判断としてのリスク管理

**経営層が「このままでは危ない」と判断するケース**

#### どんなリスクが可視化されたか

- **情報喪失リスク**：担当者退職による顧客情報・商談情報の消滅（→ Salesforce導入）
- **情報漏洩リスク**：メール誤送信・PC紛失（→ Box/Dropbox導入）
- **コンプライアンスリスク**：労基署の是正勧告・書類送検（→ クラウド勤怠導入）
- **事業継続リスク**：大手取引先からの取引停止通告（→ EDI/インフォマート導入）

#### 現場の抵抗をどう乗り越えたか

このパターンでは「経営者が決断し、現場は従う」という構造が機能する。
鍵は「リスクの金額換算」と「実際に事故が起きた他社事例の提示」である。

「このままでは○○億円の損失リスクがある」「競合のA社ではこういう事故が起きた」という形で、経営者の意思決定に必要な「数字とストーリー」を用意することが突破口になる。

#### 設計示唆

リスク可視化は「現場担当者が動けない理由」を「経営者が動く理由」に転換する最も確実な手法。プラットフォーム側がリスクレポートを提供したり、他社のトラブル事例を資料化して営業ツールにすることが有効。

---

### パターン3：競合の動向を見て

**競合他社が先行導入し、「うちも」となるケース**

#### どのタイミングで「競合が使っている」が明らかになるか

- 業界展示会・セミナーでの会話
- 取引先から「〇〇社はすでにこのシステムを使っている」という情報
- プレスリリース・導入事例の公開
- 採用要件に「Salesforce経験者」「ATS運用経験者」が増え始める

#### ファーストムーバーアドバンテージの設計

このパターンは「大企業の導入事例を公開し、業界内で話題にする」ことで加速する。
Salesforce、DocuSign、SmartHRはいずれも著名企業の導入事例をケーススタディとして積極公開し、「あの会社が使っている」という競合意識を活用した。

#### 設計示唆

業界内での「最初の10社」の導入事例を最大限に活用する。特に「競合より一歩遅れた」という心理的プレッシャーを与えるためには、競合が導入済みであることを明示するマーケティングが有効。

---

### パターン4：現場からのボトムアップ

**現場の担当者・チームが先に使い始め、組織全体に広がるケース**

#### 最初に使い始めた「チャンピオン」は誰か

- Slack：エンジニアチーム・スタートアップ出身者
- Box/Dropbox：リモートワークを先行する部門・クリエイティブチーム
- 電子契約：法務部門の若手担当者、外資系出身者

「チャンピオン」の共通特性は「業界平均より技術リテラシーが高く、前職での使用経験がある」こと。彼らが「無料プラン」「個人利用」「小規模チームでの試用」から始め、成果を見せることで組織全体への導入提案を行う。

#### 組織に広げる際の壁と突破口

| 壁 | 突破口 |
| -- | ------ |
| IT部門・情報システム部からの「セキュリティ審査未通過」という拒否 | Enterprise版・セキュリティ認証の取得により審査をクリア |
| 経営層の「費用対効果が見えない」という懸念 | チャンピオンチームでの効果計測データを先に準備 |
| 他部門の「うちには関係ない」という無関心 | 経営者を巻き込んだ全社方針化 |

#### 設計示唆

フリーミアム・無料プランは「チャンピオン育成装置」として機能する。チャンピオンが社内で成功体験を積み、経営者へ提案する流れを支援する「成功事例集・社内提案資料のテンプレート提供」がプラットフォーム側の重要な役割になる。

---

### パターン5：取引先・パートナーの引力

**取引相手が使っていることで「使わざるを得ない」ケース**

#### 誰が最初に採用し、どう引力が働いたか

このパターンはすべての事例の中で「最も強力な乗り換え圧力」を生む。

- **EDI**：大手製造業メーカーが採用 → サプライヤーへの要件化 → サプライヤーが対応 → そのサプライヤーがさらに下請けに要件化
- **インフォマート**：大手レストランチェーンが採用 → 食材卸への要件化 → 卸が飲食店に普及活動 → 飲食店もインフォマート上で検索・発注する
- **Slack Connect**：発注先や取引先からSlack Connectで招待される → 受け入れるためにSlack有料プランが必要 → 気づくと取引先とのコミュニケーションがSlack依存になる

#### 引力設計の核心

引力は「一方向の要件化」ではなく「双方向の便益」として設計されると持続する。

インフォマートが成功した理由は、買い手（飲食店）が「発注がラク」で、売り手（卸業者）が「受注処理が自動化できる」という双方向の便益を持ったことにある。一方だけが便益を得るモデルでは、便益を得ない側の参加インセンティブが弱く、普及が止まる。

## 第3部：SES業界への逆算適用

### なぜSES業界ではいまだにメール・電話が使われ続けるか

前述の5パターンを使って、SES業界に乗り換えが起きない理由を分析する。

#### パターン1（規制対応）：規制の強制力が弱い

SES業界の多重商流・偽装請負は違法リスクを内包しているが、**労働基準監督署の積極的な取り締まりが少なく、業界慣行として黙認されてきた**。
2024年のフリーランス新法は契約内容の明確化を義務付けたが、「プラットフォームを使わなければ違反」という強制力は持たない。

規制が「守れなければ免許剥奪」「取引停止」レベルに達しない限り、この圧力は弱い。

#### パターン2（リスク管理）：リスクが経営者に見えていない

SES企業経営者にとって「メール・電話で管理を続けるリスク」は可視化されていない。

- 多重商流の中で自社エンジニアがどこで何をしているか把握できていないが、「それで困ったことがない」
- 偽装請負リスクはあるが、「今まで摘発されたことがない」
- 担当営業が退職しても「人脈頼みの引き継ぎ」で何とかなってきた

**リスクが金額で語られていない**ことが、経営判断を遅らせている最大の要因。

#### パターン3（競合の動向）：業界内での事例共有が乏しい

SES業界は横の情報共有が少なく、「競合がプラットフォームを使っている」という情報が伝わりにくい。
IT展示会やセミナーで情報交換する文化が薄く、成功事例が業界内で可視化されない。

#### パターン4（ボトムアップ）：チャンピオンが育ちにくい

SES業界の現場エンジニアはプラットフォーム導入の意思決定権を持たない。
SES企業の営業担当者は「電話・メールで動いているし、それで案件が取れている」という体験を持ち、変化への動機が弱い。

試用できる無料プランや、個人レベルで試せる入口が設計されていないと、チャンピオンが生まれない。

#### パターン5（取引先の引力）：引力を生む「起点企業」がいない

**これが最も根本的な欠落である。**

インフォマート・EDIが成功したのは「大手企業が先行採用し、取引先への要件化を行った」からだ。
SES業界では、発注元の事業会社（ユーザー企業）がSES管理プラットフォームを使うインセンティブを持っておらず、引力の起点が生まれていない。

SES会社同士の水平的な関係（協力会社ネットワーク）においても、「あの会社がプラットフォームを使っているから、うちも使わないと取引できない」という状況が設計できていない。

---

### 乗り換えを起こすための「企業導入トリガー設計」

#### トリガー1：多重商流コンプライアンスリスクの可視化

**対象：SES企業の経営者**

2024年フリーランス新法・労働者派遣法の適正化圧力を背景に、「多重商流の透明化ができていないと、発注元（事業会社）からの取引停止要求を受けるリスクがある」という認識を醸成する。

具体的には、「大手事業会社が取引先SESに多重商流の開示を要求し始めている」という事例を業界内で共有し、**「うちも取引先から要求される前に対応しておくべき」という予防的行動**を引き出す。

プラットフォーム側は「商流の透明化・記録化」機能を中心に据え、**コンプライアンス対応ツールとして訴求**する。

#### トリガー2：エンジニア稼働率・案件単価の可視化による経営インパクト

**対象：SES企業の経営者・営業管理者**

「メール・電話管理では、自社エンジニアの稼働率リアルタイムが把握できない」という問題を金額換算する。

- 待機エンジニア1名あたり月70〜100万円の機会損失
- 商流ミスによる単価の10〜20%ロス
- 担当営業退職時の案件引き継ぎ失敗による売上損失

これらを「見えていないコスト」として経営者に提示する。Salesforce導入を決断させた「担当者退職による情報喪失リスク」と同じロジックを、SES文脈に翻案したアプローチである。

#### トリガー3：採用・定着におけるエンジニアの期待値活用

**対象：SES企業の経営者・採用担当**

エンジニアの転職市場では「良いSES企業かどうか」を判断する基準として、「案件の透明性」「商流の明確さ」「自分がどこで何をするか事前にわかるか」が重要視され始めている。

プラットフォームを使っている企業を「透明性が高いSES企業」としてブランディングし、**エンジニア採用の差別化要素**として訴求する。「プラットフォームに参加していない企業はエンジニアに敬遠される」という状況を作ることで、採用戦略の一環として経営者を動かす。

---

### 「取引先の引力」設計：5つのシナリオ

SES業界で「プラットフォームを使っている会社と取引したいなら、自分もプラットフォームに入らないといけない」という状態を作るための具体的設計シナリオを以下に提示する。

#### シナリオA：発注元の事業会社（ユーザー企業）を起点にする

**最も強力だが、最も難しいアプローチ**

大手事業会社（メガバンク・製造業大手・通信キャリア等）のIT調達部門に対し、**「SES取引先の多重商流を可視化・管理できるツール」**として提案し、取引条件に組み込んでもらう。

実現できれば「そのユーザー企業と取引したいSES会社はプラットフォームに参加必須」という強力な引力が生まれる。

ただし事業会社のIT調達フローは長期化するため、最初から大企業を狙うのではなく「中堅のSIer」をターゲットにすることが現実的。

**設計シナリオ：**

1. 中堅SIer（元請け）がプラットフォームを採用
2. 「協力会社（2次請け）への案件提示はプラットフォーム経由のみ」とルール化
3. 2次請けSES企業がプラットフォームに参加
4. 2次請けが3次請けとやり取りする際もプラットフォーム経由を要求
5. ドミノ的に広がる

#### シナリオB：エンジニアのポータビリティを武器にする

**ボトムアップ型の引力設計**

エンジニア個人が「自分のスキル・職歴・稼働状況」をプラットフォーム上に蓄積・管理できる仕組みを作る。エンジニアが転職・独立時に「自分のデータをそのまま持ち出せる」価値を提供する。

このとき、エンジニアが「次の会社もこのプラットフォームを使っていないと自分のデータが活きない」と感じれば、**転職先や独立後の取引先に「このプラットフォームを使ってほしい」と自発的に働きかける**。

エンジニア自身がプラットフォーム普及の営業担当になる設計。

#### シナリオC：協力会社ネットワークの可視化と信頼スコア

**信頼の非対称性を解消する引力設計**

SES業界では「どの会社が信頼できる協力会社か」の判断が困難で、新規の協力会社との取引開始に多大なコストがかかる。

プラットフォームに「協力会社の信頼スコア（過去の取引実績・稼働率・単価適正性・エンジニアの評価）」を蓄積することで、**「プラットフォーム内の企業とは安心して取引できる」という信頼の担保**を提供する。

「プラットフォーム外の企業とは取引リスクが高い」という認識が広がれば、参加していない企業は取引機会を失う。この非対称性が「参加しないと損をする」という引力を生む。

Airbnbのレビューシステム・UberのドライバーレーティングがCtoCで機能したのと同じロジックをBtoB（SES企業間）に適用する。

#### シナリオD：請求・支払いフローをプラットフォームに統合する

**金融的スイッチングコストによる引力設計**

SES取引における「請求書発行・入金確認・支払い」の一連フローをプラットフォームに内包する。

- 上流企業（発注者）がプラットフォームで支払い処理をすると、下流企業（受注者）への支払いが自動分配される
- 早期入金ファクタリング（ファイナンス機能）をプラットフォームが提供し、「プラットフォーム経由の取引は早期に入金される」という金銭的優位を設ける

請求・入金フローがプラットフォームに乗ると、参加企業は「プラットフォーム外の取引は支払いが遅くなる・管理が煩雑になる」と体験し、**全取引をプラットフォーム経由に移行するインセンティブ**が働く。

インフォマートが「年間流通金額62兆円」という規模に達した要因の一つは、受発注だけでなく請求書照合・支払い処理まで一貫したフローを提供したことにある。

#### シナリオE：独自の案件情報を取引条件にする

**情報の非対称性を活用した引力設計**

プラットフォーム上にしか存在しない「プレミアム案件」（高単価・大手エンドユーザー・長期案件）を集め、**「プラットフォームに参加しないと、その案件の情報が得られない」**という状態を作る。

これはビジネスモデルとして「プラットフォームがエージェント機能を持つ」ことを意味し、LinkedIn Premium・Indeed有料プランと同様の設計である。

SES文脈では「発注元の事業会社がプラットフォーム上に直接案件を掲載する」仕組みを作ることで、中間の元請けSIerを経由しなくても案件が取れる可能性が生まれる。これはSES企業にとって「プラットフォームに入ることで商流を短縮できる」という直接的な経済的メリットを提供する。

## 総括：SES業界に「取引先の引力」を生む優先順位

以上の分析から、SES業界マッチングプラットフォームが取るべきアクションの優先順位を以下に整理する。

| 優先度 | アクション | パターン | 時間軸 |
| ------ | ---------- | -------- | ------ |
| 最優先 | 中堅SIer（元請け）を起点にした引力設計（シナリオA） | パターン5 | 6〜18ヶ月 |
| 高 | 信頼スコア・取引実績の可視化（シナリオC） | パターン2 | 6〜12ヶ月 |
| 高 | 多重商流コンプライアンスリスクの経営者向け訴求（トリガー1） | パターン1 | 即時〜6ヶ月 |
| 中 | エンジニアポータビリティ設計（シナリオB） | パターン4 | 12〜24ヶ月 |
| 中 | 請求・支払いフロー統合（シナリオD） | パターン5 | 18〜36ヶ月 |
| 中 | プレミアム案件の独自集積（シナリオE） | パターン3 | 12〜24ヶ月 |

**今すぐ着手すべきこと：**
中堅SIer（10〜100名規模の元請け機能を持つ企業）への直販営業と、彼らの「協力会社管理コスト削減」を入口とした導入。その後、元請けの取引先（協力会社）への参加要請をプラットフォームが支援する。

**賭けを一つ選ぶとしたら：**
シナリオA（元請けSIer起点の引力設計）。これだけが「参加しないと取引できない」という真の引力を生む唯一のアプローチであり、他のシナリオは「入ってもいい」程度の動機しか生まない。

**後回しにしてよいもの：**
シナリオD（請求・支払いフロー統合）は効果が大きいが、金融規制・資金移動業登録・信用リスク管理が必要なため、規模が確立してから着手すべき領域。
