# SESプラットフォーム設計 v3補足：役割モデルの再設計

> v3の「案件企業・仲介会社・要員会社」という3分類は、実態を単純化しすぎている。
> 現実には1社が複数の役割を兼ね、さらに個人事業主という形態も存在する。
> この補足では、現実の構造に合致したプラットフォームのデータモデルと設計思想を提案する。

---

## 問題提起：なぜ「3分類」が間違いか

### 現実の多役割構造

| 企業の典型例 | 案件企業として | 仲介として | 要員会社として |
| ------------ | -------------- | ---------- | -------------- |
| 大手SIer | ◎（元請け案件を保有） | ○（SEを再委託先に紹介） | ○（自社SEを他の大手SIerに提供） |
| 中堅SIer | ◎（自社案件）かつ ○（大手の下請け） | △（社内に抱えすぎたSEを融通） | ◎（大手SIerへの要員供給） |
| SES専業中小企業 | △（自社案件を持つ場合もある） | ◎（仲介専業） | ◎（SE供給が主） |
| スタートアップ | ◎（自社プロダクト開発） | ✕ | ✕（内製志向） |
| 個人事業主SE | ✕ | △（知人を紹介する場合） | ◎（自分が稼働する） |

同一企業が「今この取引では発注側、別の取引では受注側」という状態が日常的に発生する。

### 「3分類」モデルが生む設計の歪み

- **スコアが混在する**：「良い案件企業」としての評価と「良い要員会社」としての評価が同一スコアに混入する
- **権限設計が崩れる**：「案件企業向けの機能」を「要員会社でもある企業」に見せるべきか判断できない
- **個人事業主が宙に浮く**：会社でも、かつ個人でもあるSEをどちらで扱うか定まらない

---

## 提案：「役割×トランザクション」モデル

### 基本設計思想

> **企業や個人は「プレイヤー」として単一の存在であり、各トランザクション（取引）の中で「Client（発注側）」か「Vendor（受注側）」かの役割を持つ。**

固定された会社タイプは存在しない。同じ企業が取引Aでは`Client`、取引Bでは`Vendor`になれる。

### プレイヤーの種類

```text
プレイヤー
├── 法人プレイヤー（企業アカウント）
│   ├── 大企業（従業員100名以上）
│   ├── 中小企業（従業員5〜100名）
│   └── 小規模法人（従業員1〜5名）
└── 個人プレイヤー（個人アカウント）
    ├── 個人事業主（フリーランスSE）
    ├── 副業SE（会社員兼）
    └── 法人所属SE（会社のアカウントと紐付け）
```

### トランザクションの役割定義

各取引（トランザクション）は以下の役割で構成される：

```text
トランザクション
├── Client（発注側）：エンジニアを受け入れる側
├── Vendor（受注側）：エンジニアを提供する側
└── Broker（仲介）※オプション：ClientとVendorを繋いだ仲介者
```

一社が`Client`と`Vendor`を同時に務めることはない（1トランザクション内）が、
複数のトランザクションにまたがれば両方の役割を持つ。

## スコア設計の再定義

### 役割別スコアの分離

「SESトラストスコア」をロール別に分離して蓄積する。

```text
プレイヤー XYZ株式会社のスコア
├── Clientスコア（発注側としての信頼性）
│   └── 指標：条件の後出し変更回数、支払い遅延、評価の公平性
├── Vendorスコア（受注側としての信頼性）
│   └── 指標：SE途中退場率、スキル詐称件数、契約遵守率
└── Brokerスコア（仲介としての品質）※該当する場合のみ
    └── 指標：マッチングの定着率、条件の正確な伝達率
```

これにより、「同じ会社でも案件企業としての評価は高いが、要員会社としては低い」という現実が正確に反映される。

### 多役割プレイヤーの表示例

```
XYZ株式会社
├── 登録取引数：247件
├── Clientとして：132件（スコア: 87/100）
├── Vendorとして：98件（スコア: 74/100）
└── Brokerとして：17件（スコア: 91/100）
```

## 個人プレイヤーの設計

### なぜ個人（フリーランス・個人事業主）を別モデルで扱うか

個人プレイヤーは企業と根本的に異なる：

- **キャリア資産は個人に帰属する**：企業を変えても実績が消えない
- **評判が人格に直結する**：企業評判より個人評判の方が取引判断に使われる
- **Broker機能を非公式に果たす**：「知り合いを紹介する」という個人ネットワーク

### 個人プレイヤーの機能設計

#### 個人として参加できること

- 直接Vendorとして案件を受ける（法人を介さない）
- 就業実績の第三者証明を個人名義で蓄積する
- 市場単価インデックスで自分のポジションを確認する

#### 法人所属SEとして参加できること

- 自社（法人アカウント）に紐付けたまま、個人スコアも蓄積する
- 法人を離れた後も、個人スコア・就業実績は個人アカウントに残る
  （法人アカウントからは消えるが、個人アカウントには残る）

#### 個人事業主のBroker機能

フリーランスが「知人を紹介して仲介報酬を得る」という行動を正式にサポートする：

- 「紹介リンク」を発行し、自分の紹介経由で成約した場合に手数料が入る仕組み
- 非公式な口コミ紹介を、プラットフォーム上の「透明な仲介」に変換する
- これにより、フリーランスSEがプラットフォームの伝道師になる構造が生まれる

## 多役割を前提とした機能の再設計

### ダッシュボードの二面化

企業ログイン時に「今日はClientとして使うか、Vendorとして使うか」を選ぶ（あるいは自動判定）：

- **Clientモード**：調達したいエンジニアの検索・要件登録・BPの評価閲覧
- **Vendorモード**：案件の検索・SE稼働状況の管理・Clientへの提案

同一アカウントで両モードを行き来できる設計。

### 取引ネットワークの可視化

多役割構造を見える化する「商流マップ」：

```text
A社（大手SIer）
  ↓ Client（案件を発注）
B社（中堅SIer）← Vendorとして受注
  ↓ さらにB社はClientとして発注
C社（SES専業）← Vendorとして受注
  ↓ C社のSE
D氏（個人SE）← 稼働
```

このマップを可視化することで：

- A社は「自分の案件が何次請けになっているか」を即座に把握できる
- B社は「自分はClientでもありVendorでもある」という両面の評判を管理できる
- コンプライアンス上の多重商流チェックが自動化される

### コンプライアンスチェックの多役割対応

多重商流規制のチェックは「3社の役割を固定した上で何次請けか」ではなく、「このトランザクションの連鎖が最終的に何層になっているか」を自動判定する仕組みが必要になる。

## 個人・企業混在モデルが変えるMVP設計

### 最初の参加者を「個人SE」から始める新たな正当性

役割モデルを採用すると、「個人SE」を起点にすることがより明確に正当化される：

- フリーランスSEは法人を介さずに直接参加できる
- 法人所属SEも個人アカウントで参加し、キャリア資産を積める
- SEが「紹介リンク」でBP企業を招待すれば、個人が普及の担い手になる

### 「法人が個人を招待する」引力設計

元請けSIerが「うちの協力会社はプラットフォームで管理する」と決めた際、その会社の担当PM個人にも「個人アカウント」作成を促すフローを設計する。

法人導入 → 担当者個人アカウント作成 → 個人スコアの蓄積開始 → 「このプラットフォームに自分の実績がある」という個人ロックインが同時に生まれる。

## 設計原則の更新（v3修正版）

v3の設計原則を「役割モデル」を前提として更新する：

| 旧原則（v3） | 新原則（役割モデル対応） |
| ------------ | ------------------------ |
| 「企業タイプ」ごとの価値設計 | 「役割×トランザクション」ごとの価値設計 |
| 案件企業・仲介会社・要員会社という固定分類 | Client・Vendor・Brokerという動的役割 |
| 法人スコアのみ | 法人スコア＋個人スコアの分離管理 |
| 個人SEは法人の一部 | 個人は独立したプレイヤーとして存在 |
| 企業を説得して導入させる | 個人の参加が法人の参加を引き出す |

## 残された設計課題

役割モデルを採用した場合、以下の問いに追加で答える必要がある：

1. **同一企業の役割切り替えの境界**：同じ担当者が同一プラットフォーム内でClientにもVendorにもなれる場合、利益相反（自社に有利なBPを推薦するなど）をどう防ぐか

2. **個人と法人の所属関係の変化への対応**：法人所属SEがフリーランスに転身した場合、法人アカウントと個人アカウントの分離をどう行うか

3. **Broker報酬の適法性**：個人事業主が仲介報酬を受け取る際の「二重派遣・多重商流」とならない契約設計

4. **小規模法人（1〜5名）と個人事業主の境界**：実質的に「法人化した個人」であるケースをどちらのモデルで扱うか
