# SESプラットフォーム設計 v4：3つの設計更新

> v3・役割モデル補足・ビジョン文書の検討を踏まえ、ユーザーからの3つの重要な論点——「体制で入る」モデル・複合役割ROI設計・先物マッチングリスク——を4視点並列分析し、設計に統合した文書。

---

## エグゼクティブサマリー：v4 で何が変わるか

| 更新 | v3 までの設計 | v4 での変更 |
| ---- | ------------- | ----------- |
| 役割モデル | Client / Vendor / Broker の3役割 | **Team Integrator（TI）を第4の役割として追加** |
| 参加インセンティブ | 「使えば得する」という説明 | **Broker機能はデフォルトOFF・OPT-IN設計に変更** |
| 先物マッチング | SEが主体で次案件を探す | **要員会社が主体で稼働計画を管理する設計に変更** |

## 設計更新 1：Team Integrator（体制で入るリーダー）

### なぜ既存の3役割に収まらないか

SES業界には「リーダー1名＋複数BPの体制」で案件に参加する商慣習がある。
このリーダーは：

- **Vendorとして入る**：PM・テックリードとして自身が稼働する
- **Brokerに似た機能を持つ**：BPを集めて体制を作る
- **しかしBrokerではない**：手数料を取らず、体制構築そのものが付加価値。自身が体制の一員として最後まで責任を持つ

既存のどの役割にも収まらない**第4の役割**だ。

### Team Integratorの定義

> **Team Integrator（TI）**：Client から「リーダー＋複数BP」という体制単位で発注を受け、自身がリーダーとしてVendorロールで稼働しながら、複数のSEを調達・統合して体制を組成・納品する者。体制構築の対価は仲介手数料ではなく、リーダーとしての稼働単価に内包されている。

### BrokerとTeam Integratorの決定的な違い

| 比較軸 | Broker（仲介） | Team Integrator（体制リーダー） |
| ------ | -------------- | ------------------------------- |
| 稼働の有無 | 自身は稼働しない | **自身もリーダーとして稼働する** |
| 対価の性質 | 仲介手数料（マージン） | 稼働単価（自分自身の労働対価） |
| 責任の範囲 | マッチングまで | **体制全体の成果に責任を持つ** |
| BPとの関係 | 情報を繋ぐだけ | 指揮・管理・品質保証を行う |

この区別が重要なのは、「体制で入る」は仲介業の代替ではなく、**Vendorの一形態**であるためだ。

### Team Integratorの多くは「仲介化しない」

ユーザーが指摘した通り、「体制で入る」は仲介会社の役割とは根本的に異なる。
大手SIerが「リーダー付きの体制で発注する」ケースでは：

- リーダーはまずVendorとして採用確定される
- その後リーダーがBPを選定・集める（これはリーダーの付加価値であり、仲介行為ではない）
- リーダーは体制全体の品質・成果に責任を持つ

よってTeam Integratorは「仲介業を合法化する役割」ではなく、「稼働しながら体制を組む上位Vendor」として設計する。

### プラットフォーム設計への影響

#### 体制ワークスペース機能

Team Integratorが案件に紐づけてワークスペースを作成し、以下を一体管理する：

- ロール定義（リーダー・バックエンドSE・フロントSE など）
- BP検索・オファー送信
- 体制図の可視化（Clientへの提案資料に使える）
- 三者確認フロー（Client → TI → BP の段階的承認）

#### Team Integratorスコア（TIスコア）

| 指標 | 重みづけ | 説明 |
| ---- | -------- | ---- |
| 体制完走率 | 高 | BPの途中退場なく案件を完了した割合 |
| BP定着率 | 高 | 同一TIへのBP再参加率（信頼の証） |
| 体制組成スピード | 中 | 発注から体制確定までの日数 |
| Client NPS | 高 | 案件完了時のClient評価 |
| BP側からのリーダー評価 | 中 | BPが匿名で評価するリーダーの品質 |

#### 「戻れなくなる瞬間」（Team Integrator版）

> 「体制完走率91%・BP定着率67%というTIスコアを初めて見た瞬間、スコアなしで次の体制オファーを取りに行くことが損に見える。」

#### 悪用防止：名目リーダー問題への歯止め

「リーダーとして採用されながら実際は稼働せず、BPを集めるだけ」という悪用を防ぐ：

- **稼働実績の登録必須**：TIは自身の稼働内容をプラットフォームに登録しなければならない
- **BP側の匿名評価**：「このリーダーは実際に機能していたか」をBPが評価する
- **稼働なしは自動的にBroker登録**：稼働実績がないまま体制組成だけ行うと、システムがBrokerロールに分類する

## 設計更新 2：ROI設計——「Client+Vendor > Broker損失」を実現する

### なぜこの仮説は正しく、かつ難しいか

ユーザーの仮説は正しい。

> 「案件企業・要員会社としてのメリットが仲介会社としてのデメリットを上回れば使う」

ただし実現を妨げる「タイミングのズレ」が存在する：

| 利益・損失の種類 | 発生タイミング |
| ---------------- | -------------- |
| Broker損失（情報優位の喪失） | 参加した瞬間から発生しうる |
| Vendor利益（稼働管理の効率化） | 参加初期から一定程度実現 |
| Client利益（調達品質の向上） | プラットフォームが成長してから本格実現 |

「今すぐ確実に損する」対「いつか得するかもしれない」では参加しない。

### 解決策：Broker機能はデフォルトOFF・OPT-IN設計

**設計の中心命題：**

> Broker損失の発生タイミングを遅らせ、Client/Vendor利益の実現タイミングを早める。
> 具体的には、**Broker機能は参加した機能の範囲でしか損失が発生しない**設計にする。

#### 参加初日から使える機能（Broker損失なし）

- SE稼働状況ダッシュボード（自社SEのリソース管理）
- 市場単価インデックスの閲覧
- Vendor評価の閲覧（Client側として）
- トラストスコアの段階的な蓄積開始

#### OPT-INで解放されるBroker機能

- 案件情報の公開範囲の設定
- 自社案件・自社SE情報の外部公開
- 直接マッチングの受付開始

「まずClient/Vendorとして価値を体験し、Broker機能の公開はタイミングを選べる」設計にすることで、参加障壁を下げる。

### 会社タイプ別の参加優先度と引き込み戦略

| 会社タイプ | ROI | 参加優先度 | 引き込み戦略 |
| ---------- | --- | ---------- | ------------ |
| フリーランスSE | Broker損失ゼロ・Vendor利益大 | **MVP起点** | 実績証明・単価可視化 |
| 大手SIer | Client利益大・Broker損失小 | **最優先（引力の核）** | コンプライアンス・調達効率 |
| 中堅SIer | Vendor利益大・Broker損失中 | 優先 | SE稼働率向上・単価向上 |
| SES専業中小（仲介主体） | Broker損失大 | **後回し・別設計** | 「取り残される」状況を先に作る |

### 「純粋な仲介専業」は最後でいい

SES専業中小（仲介が主収益）を最初に説得しようとしてはいけない。
彼らのROIがプラスになるのは「既存の取引先がプラットフォームに移行してから」だ。
周囲が参加してネットワーク圧力が高まった時点で「参加しないコスト」が「Broker損失」を上回る。

**仲介業者は説得で動かすのではなく、孤立コストで動かす。**

### Broker機能の「高付加価値化」：退場ではなく変容

仲介会社が「情報仲介業からコーディネーターへ進化できる」という道筋を示す：

- **複雑案件コーディネーター認定**：スキルが特殊・多国籍・規制対応など複雑な案件に特化したBrokerを「認定コーディネーター」として別枠で表示
- **BrokerコーチングスコアR**：単なる成約数ではなく「マッチングの定着率」「条件の正確な伝達率」を蓄積し、Brokerが自分の質を改善できるフィードバックを提供
- **新規SE発掘・育成Broker認定**：未経験・若手SEを発掘・育成して案件に送り出す機能。スケールしにくいが価値は高い

## 設計更新 3：先物マッチングの再設計

### ユーザーが指摘した懸念の核心

> 「先物マッチングを可能にしてしまうと、引き抜きみたいなことが横行するというツールとみられる可能性がある」

分析の結果、このリスクの本質は「実際の引き抜き被害」よりも **「業界認知リスク」** にある。

- 実際に引き抜きが起きなくても「あのプラットフォームは引き抜きに使われる」という評判で要員会社が離脱する
- SES業界のコミュニティは狭く、ネガティブな評判は1〜2ヶ月で広まる
- 要員会社（Vendor）が離脱するとプラットフォームが機能しなくなる——これが最速の崩壊経路

### 根本的な設計変更：「誰が主体か」を変える

#### 旧設計（問題あり）

```text
SEが「自分の稼働終了を登録し、次案件を探す」
→ 案件会社が稼働中のSEに直接オファー
→ 要員会社「うちのSEが引き抜かれる」
```

#### 新設計（v4）

```text
要員会社が「自社SEの稼働終了予定を登録し、次案件を取りに行く」
→ 案件会社が「タイミングが合うSEを持つ要員会社」にオファー
→ 要員会社がSEにオファーを提示・相談
→ SEが「興味あり」「辞退」を選択
→ 要員会社が案件会社に正式回答
```

#### 設計変更のポイント

| 設計軸 | 旧（SEが主体） | 新（要員会社が主体） |
| ------ | -------------- | -------------------- |
| 登録者 | SE個人 | 要員会社（自社SEの情報を登録） |
| オファーの宛先 | SE個人に直接 | 要員会社に送る |
| SEの役割 | 自分で動いて次を決める | 「興味あり」を選ぶ。交渉は要員会社 |
| 要員会社の立場 | 引き抜かれる側 | **先手を打って次案件を確保する側** |

### 「引き抜き防止インフラ」として要員会社に売る

設計思想を逆転させる：

> 「このプラットフォームの先物マッチングを使うほど、自社のSEが他社に引き抜かれにくくなる」

要員会社にとっての「戻れなくなる瞬間」：

> 「自社SEの案件終了2ヶ月前に次案件の内定を自社経由で確保したとき、その体験を一度でもすると、バタバタと案件終了直前に動き回ることが耐えられなくなる。」

### 情報開示の範囲設計

| 誰に | 開示する情報 | 開示しない情報 |
| ---- | ------------ | -------------- |
| SE | 自分へのオファー数・案件概要・自社経由の交渉状況 | 案件会社名（交渉前）・他SEへのオファー |
| 要員会社 | 自社SE全員の先物状況・オファー件数 | 他社SEへのオファー状況 |
| 案件会社 | 条件マッチするSEの数・要員会社との交渉状況 | 他案件会社のオファー状況 |

SEには「3社がオファーを出しています」という**市場価値の可視化**は提供する。しかし承諾・交渉は要員会社主導にする。これによりSEが「自社の価値が見える」体験を維持しながら、要員会社が「うちが守っている」と感じられる設計を両立する。

## 批判的検討への応答

4視点の分析から浮上した批判のうち、特に重要な3つに応答する。

### 批判1：「良い案件・良い要員が来ない」ニワトリと卵問題

**デビルズアドボケートの指摘：**
大手SIerが参加しても「既存の人脈で埋まらなかった余り物案件」しか出さない可能性がある。

**v4 での対応：**

まず「良い案件が来なくても価値のある機能」から始める。
フリーランスSEはプラットフォームに参加するだけで市場価値の可視化・実績証明の蓄積が始まる。案件の質に依存しない価値を先行させる。

大手SIerに対しては「調達機能」ではなく「コンプライアンス管理機能」を入口にする。多重商流の自動チェック・取引先のスコア管理は、案件の流通量に関係なく初日から価値がある。

「余り物しか出さない」問題は、**使う機能をClient/Vendor管理機能に限定した初期参加**で解消する。案件の公開はOPT-INであるため、案件企業は最初に「管理機能の利用者」として参加し、価値を実感してから案件を出す設計にする。

### 批判2：「目的（仲介退出）と設計（仲介にも優しい）の矛盾」

**デビルズアドボケートの指摘：**
仲介会社に優しい設計では多重商流の解消にならない。「デジタル化された多重商流」になるだけだ。

**v4 での応答：**

「仲介業者の退出」を目的に掲げることをやめる。代わりに「**役割の自然な変容**」を設計する。

仲介が持つ価値の本質は「コーディネート能力」だ。情報仲介（情報の非対称性で食う）はプラットフォームに代替されるが、複雑な調整・トラブル対応・特殊スキルの発掘といった能力は代替されない。プラットフォームはその「残る価値」が発揮できる舞台を用意し、情報仲介業者が自然にコーディネーターへ変容する経路を設ける。

結果として：

- 低付加価値の情報仲介は「使われなくなって自然退場」する
- 高付加価値のコーディネーターは「プラットフォームの中で生き残る」
- プラットフォームが仲介を「排除」するのではなく、市場が「選別」する

### 批判3：「トラストスコアのゲーミング」

**デビルズアドボケートの指摘：**
スコアが可視化されると相互高評価談合・名義変更による再登録・信頼できる相手との取引だけ記録する、などの操作が起きる。

**v4 での対応：**

スコアの公開順序を変える：

1. **フェーズ1**：スコアは「自社だけが見える」個社フィードバック機能として開始。「あなたはこの指標が弱い」という改善ガイドとして機能させる
2. **フェーズ2**：スコアを「取引相手にのみ見える」設計に移行（取引前に確認できる）
3. **フェーズ3**：十分なデータ蓄積後に「業界統計としての開示」へ移行

また、スコアを「比較・ランキング」ではなく「取引判断の参考情報」として設計することで、ゲーミングの動機を下げる。「スコア1位になる」ではなく「最低限の信頼基準を満たす」という性格にする。

## 設計原則の更新（v4版）

| v3 までの原則 | v4 での更新 |
| ------------- | ----------- |
| Client / Vendor / Broker の3役割 | **+ Team Integrator（体制で入るリーダー）** |
| 参加すれば全機能が使える | **Broker機能はOPT-IN。Client/Vendor機能から始める** |
| 仲介業者を最終的に排除する | **情報仲介は代替し、コーディネーターへの変容を支援する** |
| SEが主体で先物マッチングを使う | **要員会社が主体で稼働計画を管理し、先手を打つ** |
| スコアを公開して透明性を確保 | **スコアは個社フィードバックから段階的に公開する** |

## MVP v4 への影響

v4 の設計更新を受けて、MVPで最初に作るべきものの優先順位が変わる。

### 最初の6ヶ月でつくるもの

| 機能 | 対象ユーザー | 理由 |
| ---- | ------------ | ---- |
| SE個人の実績証明・キャリア資産 | フリーランスSE | 案件流通量に依存しない価値。MVP起点 |
| 要員会社のSE稼働管理ダッシュボード | 中堅・SES専業 | Broker機能なし。参加障壁が低い |
| 大手SIerの取引先コンプライアンス管理 | 大手SIer担当PM | 案件公開なし。管理機能として入口化 |
| Team Integratorの登録・TIスコア基盤 | フリーランスPM・中堅SES企業PM | 体制入りモデルの透明化 |

### あえて最初に作らないもの

- 先物マッチングの「予約・内定」機能（要員会社主導の設計が固まってから）
- Broker向け仲介機能（OPT-IN設計でフェーズ2以降）
- スコアの公開・比較機能（フェーズ2以降）

## 残された設計課題

1. **Team Integratorの認定基準**：誰でもTIを名乗れる状態をどう防ぐか。プラットフォーム側が最初のTI認定を手動で行う「ドゥ・シングス・ザット・ドント・スケール」期間が必要

2. **要員会社主導の先物マッチングにおけるSEの離脱意思への対応**：SEが「要員会社を変えたい」と思っている場合、要員会社主導設計がSEを閉じ込めるツールになりかねない。SEが「自社を経由しないで動く」選択肢を維持しながら、要員会社も守れるバランス設計が必要

3. **Team IntegratorとBrokerの実務的な境界**：中堅SIerのPMが「自分はTIだ」と言い張りながら実態はBroker的行動をとるケースの検出
