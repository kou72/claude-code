# 「体制で入る」モデルの役割設計：分析レポート

> 本レポートは ses_platform_design_v2.md の3プレイヤーモデル（Client/Vendor/Broker）を前提として、「体制で入るリーダー」という商慣習をどう定式化し、プラットフォームに組み込むかを詳細に分析する。

---

## 1. 役割定義の問題：「体制リーダー」とは何者か

### 1-1. 現行の3プレイヤーモデルとの照合

現行設計では、SES業界の参加者を以下3つの動的な役割で定義している。

| 役割 | 本質 | 典型例 |
| ---- | ---- | ------ |
| Client | 人材・体制を調達する側 | 大手SIer、事業会社 |
| Vendor | 人材・稼働を提供する側 | SES企業、フリーランス |
| Broker | 需給を仲介する側 | 仲介会社、エージェント |

「体制で入るリーダー」は、この3役割のどれにも単純に当てはまらない。

- **Vendorとして入る**：リーダー本人がPM・テックリードとして稼働している点でVendorだ
- **Brokerに似た機能を持つ**：BPを集める行動は仲介と類似している
- **しかしBrokerではない**：手数料を取らず、体制構築そのものがリーダーの付加価値であり、自分自身が体制の一員として稼働する

この「自分も稼働しながら、体制を組む人間を引き連れてくる者」は、既存のどの役割にも収まらない**第4の役割**だ。

### 1-2. 「体制リーダー」の定義

以下のように定義する。

> **体制リーダー（Team Integrator）**：Client（大手SIer等）から「リーダー＋複数BP」という体制単位で発注を受け、自身がリーダーとしてVendorロールで稼働しながら、複数のSE（BP）を調達・統合して体制を組成・納品する者。体制構築の対価は仲介手数料ではなく、リーダーとしての稼働単価に組み込まれている。

### 1-3. BrokerとTeam Integratorの決定的な違い

この区別はプラットフォーム設計に直結するため、精密に整理する。

| 比較軸 | Broker（仲介） | Team Integrator（体制リーダー） |
| ------ | -------------- | ------------------------------- |
| 稼働の有無 | 自身は稼働しない（情報を繋ぐ） | 自身もリーダーとして稼働する |
| 対価の性質 | 仲介手数料（マージン） | 稼働単価（自分自身の労働対価） |
| 責任の範囲 | マッチングまで（成約後は免責が多い） | 体制全体の成果に責任を持つ |
| BPとの関係 | 情報的な仲介（会わせるだけ） | 指揮・管理・品質保証を行う |
| リスクの取り方 | 仲介失敗のリスクのみ | 体制全体が機能しない場合のリスクを取る |
| 発注者との関係 | 要員供給の手配人 | プロジェクトの責任者 |

これを別の言い方で表すと：

- Brokerは「繋いだら終わり」
- Team Integratorは「繋いだ後も自分が中心にいる」

この違いは、プラットフォーム上でのスコア設計・機能要件・課金構造に根本的な影響を与える。

### 1-4. 多くのプレイヤーが持つ複合的な役割マップ

SES業界では「1社が複数役割を兼ねる」ことが常態だ。Team Integratorが加わると、実態はより複雑になる。

```text
【複合プレイヤーの典型パターン】

パターンA：大手SIer
  - 自社案件では → Client
  - 協力会社に体制発注するとき → Client（Team Integrator指名あり）

パターンB：中堅SES企業のPM
  - 大手案件に入るとき → Team Integrator（自社SEを連れていく）
  - 自社の下請けを集めるとき → Broker的行動も含む

パターンC：フリーランスPM
  - 自分の稼働のみ → Vendor
  - BPネットワークを持つ場合 → Team Integrator

パターンD：仲介会社
  - 通常 → Broker
  - 体制提案をする場合 → Team Integrator的行動
```

## 2. プラットフォームでの設計

### 2-1. 「体制案件」という新しい案件タイプ

現行の案件登録モデルは「1名の要員を探す」という前提で設計されている可能性が高い。「体制で入る」商慣習を取り込むには、**案件タイプを2種類に分ける**ことが出発点だ。

#### 案件タイプ1：単体要員案件（Individual）

既存設計の延長。1名のSEやフリーランスを探す案件。

#### 案件タイプ2：体制案件（Team Formation Request）

以下の情報を含む。

```text
体制案件の記述フォーマット（案）

案件名：[プロジェクト名]
案件種別：体制案件
体制の構成：
  - リーダー：1名（PM経験5年以上・金融系案件経験必須）
  - バックエンドSE：2名（Java・SpringBoot 3年以上）
  - インフラSE：1名（AWS・オンプレ混在環境経験）
体制規模：4名
参画形態：リーダー同行型（リーダーがBPを連れて参画する形を想定）
リーダー先行面談：あり（リーダーのみ先に面談→承認後にBPの面談調整）
リーダー条件：Team Integrator実績スコア XX以上推奨
```

このフォーマットにより、Clientは「体制を発注したい」という意図を明示できる。

### 2-2. Team Integratorがプラットフォームでやること

体制リーダーがプラットフォームを使ってBPを集める場合の機能要件を整理する。

#### 機能1：体制編成ワークスペース

Team Integratorが体制案件を受注（あるいは受注見込み）した後、BPを集めるための専用ワークスペース。

```text
体制ワークスペースの機能

[ワークスペース作成]
  - 案件IDに紐づけて作成
  - 必要なロール・スキル・稼働条件を定義

[BP検索・招待]
  - 条件検索（スキル・稼働可能時期・過去の体制参画実績）
  - 候補者にプラットフォーム内でオファーを送信
  - オファー文に「この案件の概要」「報酬レンジ」「体制内でのロール」を明示

[体制の可視化]
  - ロール別のステータス管理（未充足→打診中→内定→確定）
  - 体制図のリアルタイム更新
  - Clientへの体制提案資料（PDF）の自動生成

[BP側への情報提供]
  - Team IntegratorのTI実績スコア（過去の体制での定着率・完走率）を閲覧可能
  - 「このリーダーが組んだ過去の体制の平均稼働期間」が見える
```

この機能の核心は「BPがリーダーの実績を見て判断できる」点だ。現状は「知り合いのリーダーから声がかかるかどうか」という属人的な繋がりだけで決まっている。

#### 機能2：体制内コミュニケーション

体制案件の参画中、Team IntegratorとBP群の間の情報共有チャンネル。参画前（体制組成段階）・参画中・参画後（実績記録）を通じて一つのワークスペースで管理する。

#### 機能3：体制案件の先物予約との連携

existing の「先物マッチング」（打ち手2）と組み合わせる。

```text
【体制先物の流れ】

1. Team Integratorが「3ヶ月後の体制案件の見込み」をワークスペースに登録
2. 必要なロール・スキルをプラットフォームが自動でBP候補に通知
3. BPは「3ヶ月後の稼働予定あり」で仮参画意思表示
4. Team Integratorが体制を仮組み→ClientにPre-確定体制として提案可能
```

これにより、Team Integratorは「体制が整ってから発注を取りに行く」という逆順（現状は発注後にバタバタ集める）が可能になる。

### 2-3. 体制案件の面談フロー設計

体制案件には「リーダー先行面談」という業界特有の慣習がある。これをプラットフォームで正式サポートする。

```text
【体制案件の面談フロー】

Stage 1：リーダー面談
  - ClientがTeam Integratorを面談
  - 合否をプラットフォームで記録
  - 合格→Stage 2に進む権限がTeam Integratorに付与される

Stage 2：BP面談調整
  - Team IntegratorがワークスペースでBPを指名
  - Clientとの面談日程をプラットフォームが自動調整
  - 面談ごとに合否を記録

Stage 3：体制確定・参画開始
  - 全ロールが充足→体制確定としてClientが承認
  - 三者承認（Client・Team Integrator・各BP）で実績記録が確定
```

## 3. Team Integratorのスコア設計

### 3-1. 既存スコア設計との違い

現行の ses_platform_design_v2.md では以下のスコアを設計している。

- PMの「採用精度スコア」
- 仲介営業の「価値証明スコア（定着率・面談通過率）」
- SEの「就業実績証明（第三者証明）」

Team Integratorは「体制を組む側」として、これらとは異なる評価軸が必要だ。

### 3-2. TI実績スコア（Team Integrator Score）の定義

以下のデータポイントから構成する複合スコア。

#### データポイント1：体制完走率

組んだ体制がプロジェクト期間を完走したかどうか。

```text
体制完走率 = 体制内BP全員が契約満了まで稼働した案件数 / 体制案件総数

※ 途中退場者が出た場合の代替調達の速度・成否も加点/減点要素
```

これは「リーダーの人マネジメント力・案件見極め力」を示す最も重要な指標だ。

#### データポイント2：BP定着率（BP Retention Rate）

過去に体制を組んだBPが、次回もそのTeam Integratorの声かけに応じるかどうか。

```text
BP定着率 = 複数回同じリーダーの体制に参加したBP数 / 過去に体制参加したBP総数
```

「優秀なリーダーには人が集まる」という業界の常識を数値化する。

#### データポイント3：体制組成スピード

案件発注から体制確定まで何日かかったか。

```text
体制組成スピード = 体制ワークスペース作成から全ロール確定までの平均日数
```

「この人に頼むとすぐ集まる」という評判を定量化する。

#### データポイント4：ClientからのNPS（体制品質スコア）

体制完了後にClientが回答する。「このTeam Integratorが組んだ体制に、また依頼したいか」を10点満点で評価。

#### データポイント5：BP側からのリーダー評価

体制参加後にBPが評価する。「このリーダーと再度一緒に働きたいか」「指示の明確さ・案件情報の透明性」。

### 3-3. TIスコアの表示設計

```text
【TI実績スコードの表示例】

Team Integrator：田中 太郎
TIスコア：★★★★☆（4.3/5.0）

実績サマリー：
- 体制案件実績：12件
- 体制完走率：91%（業界平均 74%）
- BP定着率：67%（同一リーダーに複数回参加したBP）
- 平均体制組成日数：8.2日
- Client NPS：8.4/10

専門体制タイプ：
- 金融系基盤案件（最多）
- 規模：3〜6名体制
- 参画期間：平均14ヶ月

一緒に働いたBPのスキルタグ：
  Java / SpringBoot / Oracle / 金融システム / 首都圏
```

このスコアが機能することで、以下が実現する。

- **Clientにとって**：「このリーダーが組んだ体制は完走率が高い」という根拠に基づく選択が可能
- **BPにとって**：「このリーダーに参加すると案件が安定する」という情報で意思決定ができる
- **Team Integratorにとって**：体制構築の実績が「見えない資産」から「可視化された競争優位」になる

## 4. 大手SIerが「体制で入る」要件を出す場合の表現設計

### 4-1. 現状の問題

大手SIerが「リーダー付きの体制が欲しい」という発注をする際、現状は以下のような非構造化された依頼になっている。

```text
「PM1名と、その方が連れてこられるJavaのメンバーを2名お願いしたいんですが、
 いつものAさんみたいな感じで頼める方いますか？」（口頭 or メール）
```

これをプラットフォームで構造化する。

### 4-2. 体制要件の記述フォーマット

```text
【体制案件の登録フォーム設計】

案件種別：[個別要員 / 体制] → 「体制」を選択

── 体制要件 ──

リーダー要件：
  - PM / テックリード / アーキテクト（いずれか選択）
  - 必須スキル（フリーテキスト）
  - 必須経験：体制リーダー経験 [あり必須 / あれば尚可 / 不問]
  - TIスコア要件：[ X以上推奨 / 指定なし ]

BP要件（ロールごとに追加）：
  ロール1：バックエンドSE × 2名
    必須スキル：Java / SpringBoot 3年以上
    歓迎スキル：金融系案件経験
  ロール2：インフラSE × 1名
    必須スキル：AWS / オンプレ
    ----（追加ボタンで任意数）

体制組成の方式：
  [A] リーダーが自身のネットワークでBPを集める（自己組成型）
  [B] プラットフォームのマッチングでBPを集める（プラットフォーム組成型）
  [C] リーダー指名なし（プラットフォームが体制を自動提案）

リーダー先行面談：[必要 / 体制一括面談]
```

### 4-3. Clientへの体制提案の受け取り方

体制案件の登録後、プラットフォームは以下を行う。

```text
[自己組成型（A）の場合]
  - 条件に合致するTIスコアを持つTeam Integratorに通知
  - 関心があるTeam Integratorが体制ワークスペースを作成→仮体制を組んで提案
  - ClientはTIスコア・仮体制の顔ぶれを見て面談実施を判断

[プラットフォーム組成型（B）の場合]
  - 条件に合致するリーダー候補とBP候補を自動マッチング
  - 「このリーダー + このBP構成」という体制案をプラットフォームが複数提示
  - ClientとTeam Integratorがプラットフォーム上で体制を確定

[体制自動提案型（C）の場合]
  - プラットフォームが過去の共同実績・相性スコアを元に体制案を自動生成
  - Clientは提案された体制案を一括承認 or 調整
```

## 5. 仲介との違いがプラットフォーム設計に与える影響

### 5-1. 課金モデルへの影響

現行の課金設計は「面談成立課金」が軸だ。体制案件の場合、以下の問題が生じる。

| 課金の問題 | 詳細 |
| ---------- | ---- |
| 面談回数が多い | 体制案件ではリーダー面談 + BP面談×N回が発生する |
| 誰が課金対象か | Team Integratorが課金を払うのか、Clientが払うのか |
| 価値の大きさが違う | 4名体制の成立は1名成立より価値が大きい |

体制案件の課金モデル（案）：

```text
体制案件専用の課金設計：

オプションA：体制確定成功報酬型
  体制が確定した時点で、Client側に「体制確定料」を課金
  金額：1体制あたり 20,000〜50,000円（体制規模に応じてスケール）

オプションB：リーダー面談無料 + BP面談一括課金
  リーダー面談：無料（入口のハードルを下げる）
  体制内BP面談：1名あたり 3,000円（まとめて請求）

推奨：オプションA + ワークスペース使用料（月額）の組み合わせ
  - 体制ワークスペース：月額 9,800円 / ワークスペース（Team Integrator側が負担）
  - 体制確定料：30,000円（Client側が負担）
```

### 5-2. スコア連携への影響

Brokerの「仲介スコア（定着率・面談通過率）」とTeam Integratorの「TIスコア（体制完走率・BP定着率）」は、同一ダッシュボードに並べて表示してはいけない。**異なる評価軸だからだ。**

```text
【プロフィールページの役割ごとの表示設計】

ユーザーが複数役割を持つ場合：

  [Vendorとしての実績]
    就業実績証明：XX件（三者承認済み）
    スキルタグ：Java / AWS / 金融

  [Team Integratorとしての実績]
    体制案件実績：12件
    体制完走率：91%
    TIスコア：4.3/5.0

  [Brokerとしての実績]（仲介行為がある場合）
    仲介成約数：XX件
    仲介定着率：業界平均比 +34%
    仲介スコア：4.1/5.0
```

この「役割ごとの実績分離」こそが、プラットフォームが複合プレイヤーを正しく扱える証明になる。

### 5-3. Team Integratorが「仲介化」することへの警戒と歯止め

Team Integratorが実際には稼働せず、体制を組んで手数料だけ取る「名目リーダー」問題が現実に存在する。

```text
【名目リーダー問題の実態】
  - リーダーが体制を組んで発注を取る
  - 実際にはほとんど現場に入らず、BPに任せきりにする
  - リーダーとしての単価をBPのマージンから搾取している実態

【プラットフォームでの歯止め設計】

  歯止め1：稼働実績の記録
    - 体制内でのリーダー自身の稼働時間・参画継続性を記録
    - 「リーダーが途中離脱した」という事実がBP・Client双方から記録可能

  歯止め2：BP側からのリーダー評価
    - 「リーダーが現場にいたか」「指示が機能していたか」を匿名評価
    - この評価がTIスコアに反映され、名目リーダーは低スコアになる

  歯止め3：役割の明示的な分離
    - プラットフォーム上でTeam Integratorは「自身の稼働内容」を登録必須にする
    - 「稼働なし体制組成者」はBrokerとして登録させる（別課金・別スコア体系）
```

## 6. BP選定の透明化

### 6-1. 現状の不透明性の構造

現状のBP選定は以下のメカニズムで動いている。

```text
【現状の不透明な選定プロセス】

Team Integrator（リーダー）の頭の中：
  ↓「声かけ順」
  1. 一番仲の良い人（信頼 or 義理）
  2. 前回一緒だった人（馴染みやすさ）
  3. 困ってる会社の人（恩着せ or 人情）
  4. 条件に合う人（スキルは最後に考える）

→ ClientにとってもBPにとっても、なぜ自分が選ばれた（選ばれなかった）か不明
```

### 6-2. プラットフォームによる透明化・標準化の設計

#### 透明化施策1：選定理由の構造化記録

Team Integratorが体制ワークスペースでBPにオファーを送る際、以下の選定理由を記録する（BP本人と、最終的にはClientが閲覧可能）。

```text
オファー送信フォーム：

対象者：[SE名]
ロール：バックエンドSE
選定理由（構造化）：
  ✓ スキル適合：Java 8年・SpringBoot 4年（要件：3年以上）
  ✓ 過去同一リーダー実績：2回（体制完走）
  ✓ 業種適合：金融系案件経験あり
  ✓ 稼働可能時期：要件と一致

追加コメント（任意）：フリーテキスト
```

この記録により：

- BPは「なぜ声がかかったか」が分かる（スキルの市場価値が分かる）
- 選ばれなかったBPに「なぜ選ばれなかったか」を間接的に示すことができる
- Clientは「このリーダーがなぜこのBP構成にしたか」の根拠を確認できる

#### 透明化施策2：BP側の「体制参加履歴」可視化

BPのプロフィールに「過去参加した体制の情報」が蓄積される。

```text
【BPプロフィールの体制参加履歴】

就業実績（体制案件）：
  - [案件A] Team Integrator：田中太郎 / 役割：バックエンドSE / 期間：14ヶ月 / 完走 ✓
  - [案件B] Team Integrator：佐藤花子 / 役割：インフラSE / 期間：6ヶ月 / 完走 ✓
```

「どのリーダーの体制に参加してきたか」がBPの実績の一部になる。良いリーダーの体制に何度も呼ばれることが、BPとしての信頼の証明になる設計だ。

#### 透明化施策3：体制オファーの競争原理の導入

Team Integratorが複数のBPに対して同じロールでオファーを出す場合、BPはそのオファーを受けるかどうかを判断できる。

```text
【BPへのオファー通知の設計】

「田中太郎さんから体制案件のオファーが届きました」

  案件概要：[概要]
  あなたのロール：バックエンドSE
  期間：6ヶ月（延長あり）
  参画場所：首都圏（週4日常駐）
  TIスコア：4.3/5.0（体制完走率91%）

  [承諾する] [辞退する] [詳細を確認する]
```

BPが辞退できるという設計が重要だ。「声がかかったら断れない」という現状の力学を壊し、**BPが主体的に体制を選べる**状態にする。これが「BPのインサイト（自分の市場価値を知りたい）」への答えになる。

## 7. 「戻れなくなる瞬間」の追加設計（体制リーダー向け）

既存の3プレイヤーの「戻れなくなる瞬間」に、Team Integratorとしての瞬間を加える。

| プレイヤー | 戻れなくなる瞬間 | 戻ることが恥ずかしい理由 |
| ---------- | ---------------- | ------------------------ |
| Team Integrator | 体制完走率91%・BP定着率67%という自分のTIスコアを初めて見た瞬間 | この数字なしで次の体制オファーを取りに行くことが怖くなる |
| BP（体制参加者） | 「過去3つの体制案件のリーダーからの評価」が就業実績証明に記録された瞬間 | スキルシート1枚だけで人材市場に出ることが貧しく見える |
| Client（体制発注側） | 「A社のリーダーが組んだ体制は完走率が高い」という比較データを初めて見た瞬間 | 勘と義理だけで体制リーダーを選び続けることへの不安が生まれる |

## 8. まとめ：設計への統合方針

### 8-1. 既存設計との整合

本分析は ses_platform_design_v2.md の5つの打ち手と整合している。

| 既存の打ち手 | 体制リーダー設計との接続 |
| ------------ | ------------------------ |
| 打ち手1：「判断の証拠」エンジン | TIスコア（体制完走率）がClientの体制発注判断の「証拠」になる |
| 打ち手2：先物マッチング | 体制先物（3ヶ月後の体制を今から仮組みする）として拡張 |
| 打ち手3：就業実績の第三者証明 | 体制参加実績も三者承認でSEの実績証明に組み込む |
| 打ち手4：仲介営業の「価値証明スコア」 | TIスコアはBrokerスコアとは別軸で設計・表示する |
| 打ち手5：招待制クローズドコミュニティ | TIスコアが一定以上のTeam Integratorのみ体制案件を掲載できる審査を追加 |

### 8-2. 追加すべき開発優先度

| 機能 | 優先度 | 理由 |
| ---- | ------ | ---- |
| 体制案件タイプの追加 | 高 | 体制商慣習を扱う上での前提条件 |
| 体制ワークスペース | 高 | Team Integratorがプラットフォームを使う主要動機 |
| TIスコアの定義・計算ロジック | 高 | 「戻れなくなる瞬間」の設計の核心 |
| BP側オファー受諾フロー | 中 | 透明化の鍵だが、MVP段階では簡易実装でも可 |
| 体制案件の課金設計の確定 | 中 | 収益モデルの整合確認 |
| 名目リーダー歯止め機能 | 低（後続フェーズ） | 初期は信頼できるTIのみ招待制で対処 |

### 8-3. 最終的な役割マップ（4役割モデル）

```text
【SESプラットフォームの4役割モデル】

Client（発注側）
  └ 単体要員を調達する
  └ 体制を発注する（Team Integratorを指名 or プラットフォームに提案依頼）

Vendor（稼働側）
  └ 単体で案件に参画する
  └ 体制の一員（BP）として参画する

Team Integrator（体制組成側）
  └ Vendorとして自身も稼働しながら
  └ 複数BPを調達・統合して体制を納品する

Broker（仲介側）
  └ 需給情報を繋ぐ
  └ 稼働せず、情報の橋渡しで価値を出す

※ 1社・1人が複数役割を同時に持つことが常態
※ Team IntegratorはVendorの特殊形として定義（稼働を伴う点でBrokerと区別）
```

## 付録：体制リーダーの「言えない本音」インサイト

体制リーダーは現状のSESプラットフォーム設計では最も考慮されていないプレイヤーだ。その最深インサイトを整理する。

#### 表面の不満

「良いBPがいない」「いつもの人が空いていない」「急に断られると困る」。

#### 隠れたペイン

体制を組む労力は「無償で行っている」ことへの不満。受注できなかった案件でもBP探しの時間は消費される。「体制構築力」という高度なスキルが、単なる「コネを持っている人」として扱われている。

#### 最深インサイト

> 「自分が連れてくる人の質が、自分の評判そのものだ。でも、それが誰にも見えない。」

体制リーダーは自分のネットワーク・人を見る目・プロジェクト完走能力によって、実際には高い価値を生み出している。しかしその価値は「口コミと義理」の世界でしか可視化されない。スコアとして記録され、次の発注の根拠になることが、体制リーダーが最も必要としているものだ。

> 「自分のTIスコアが4.3/5.0だと分かった瞬間、次の体制オファーをスコアなしで取りに行くことが損に見える。」

この文が書けた。Team Integratorは「戻れなくなる」設計の対象として有効なプレイヤーだ。
