# SESプラットフォーム：構造整理

> プレイヤー → 役割 → 複合 → 痛み・ニーズ → 現状での解消方法 → プラットフォーム機能 → ゴール
> という順序で、設計の根拠を積み上げる文書。

---

## 1. プレイヤーの整理

### 法人

| プレイヤー | 典型例 |
| ---------- | ------ |
| 大手SIer | 元請け案件を保有し、外部BPを大量調達する |
| 中堅SIer | 案件の受注と要員の供給を両方行う |
| SES専業会社 | SEの供給・仲介が主事業 |
| 事業会社・スタートアップ | 自社プロダクト開発のためにSEを調達する |

### 個人

| プレイヤー | 典型例 |
| ---------- | ------ |
| 法人所属SE | 要員会社に在籍し、客先常駐で稼働する |
| フリーランスSE | 個人事業主として直接または仲介経由で案件を受ける |
| PM・テックリード | 体制リーダーとして入り、BPを束ねる |

---

## 2. 役割の整理

1社・1人が複数の役割を担うことが前提。役割は固定ではなく「この取引においてどの立場か」で決まる。

| 役割 | 定義 |
| ---- | ---- |
| **Client（案件）** | エンジニアを調達・活用する側 |
| **Vendor（要員）** | エンジニアを提供・稼働させる側 |
| **Broker（仲介）** | ClientとVendorを繋ぎ、単価差分（マージン）を収益とする側 |
| **Team Integrator（TI）** | Vendorとして自身も稼働しながら、自社を経由してBPを調達・管理し、BP部分のマージンも収益とする体制リーダー |

---

## 3. 複合プレイヤーの実態

ほとんどの法人は複数の役割を同時に担っている。

```text
大手SIer
  → 大手メーカーから案件受注（Vendor）
  → 協力会社へBPを発注（Client）
  → 一部案件は仲介会社として流通（Broker）

中堅SIer
  → 大手SIerへSE供給（Vendor）
  → 自社案件に外部BPを入れる（Client）
  → 余剰SEを他社に融通（Broker）

フリーランスPM
  → 自分が稼働（Vendor）
  → BPを集めて体制を作る案件では（Team Integrator）
```

**設計上の含意**：「この会社は仲介会社だ」という固定分類は現実に合わない。
同一企業でも取引によって異なる役割を持ち、それぞれの役割ごとに評価・機能・ROIが異なる。

## 4. 各役割プレイヤーの痛み・ニーズ・インサイト

### Client（案件企業）

| 層 | 内容 |
| -- | ---- |
| 表面の痛み | 良いエンジニアが見つからない。スキルシートと実態が乖離している |
| 隠れたペイン | 仲介を重ねるほど単価が上がり予算が圧迫される。担当者が変わると関係がリセットされる。多重商流でコンプライアンスリスクが見えない |
| 最深部のインサイト | 「良いエンジニアとの継続的な関係」を欲している。仲介会社に情報を握られている構造への不満。「今稼働しているSEが実は何次請けか分からない」という不安と法的リスク |

### Broker（仲介会社）

| 層 | 内容 |
| -- | ---- |
| 表面の痛み | 良い要員が足りない。案件の条件が後出し変更される |
| 隠れたペイン | 情報仲介だけでは価値が薄く、いつでも外される恐怖。大手SIerとの力関係で単価を叩かれる |
| 最深部のインサイト | 「情報・人脈」という自社の価値がいつ陳腐化するか分からない根源的な不安。あえて不透明な構造を維持することで「自分がいないと困る」状態を作らないと生き残れない |

### Vendor（要員会社）

| 層 | 内容 |
| -- | ---- |
| 表面の痛み | SE稼働率が安定しない（ベンチ期間の損失）。仲介に単価を抜かれSEへの支払いが低くなる |
| 隠れたペイン | SEが案件先に引き抜かれる。SEの実績・スキルが会社に蓄積されない（個人が辞めると消える）。良い案件は仲介会社経由でしか来ない |
| 最深部のインサイト | 「自社SEの市場価値を自分でコントロールできない」という無力感。SEが離れると収益が直撃するのに、SEを引き止める合理的な手段が乏しい |

### 個人SE

| 層 | 内容 |
| -- | ---- |
| 表面の痛み | 単価が低い。スキルに見合った案件に入れない。次の案件が不安 |
| 隠れたペイン | 実績・評価が転職や独立の際にリセットされる。仲介が増えるほど自分の手取りが減るが、その構造が見えない |
| 最深部のインサイト | 「自分の市場価値を客観的に知りたい・証明したい」という欲求。仲介を通さず直接取引できれば単価が上がるのに、という閉塞感 |

### Team Integrator（体制リーダー）

| 層 | 内容 |
| -- | ---- |
| 表面の痛み | 良いBPが見つからない・集まらない。BPの質を事前に判断できない |
| 隠れたペイン | 体制構築能力が客観的に証明されない。良いBPとの関係が個人の人脈に依存するため、人脈が切れると体制が組めない |
| 最深部のインサイト | 「自分の実績（体制完走率・BP定着率）を見えるカタチで持ちたい」。BPとの信頼関係がプラットフォームに蓄積されれば、人脈ではなく実績で体制を組める |

## 5. 現状（メール・多重下請け構造）でのニーズ解消の実態

現在の構造がどう機能し、何を犠牲にしているか。

### Client

| ニーズ | 現状の解消方法 | 犠牲にしていること |
| ------ | -------------- | ------------------ |
| 良いエンジニアを探す | 複数仲介会社に一斉メール。量でカバー | 調達に時間がかかる。単価が上がる |
| スキルと実態の確認 | 面談で見極める | PMの時間を大量消費。精度は低い |
| コンプライアンス担保 | 書類を積み上げる | 実態の多重商流は見えない。書類だけの対応 |
| 担当者変更時の引き継ぎ | 引き継ぎメール・口頭説明 | 文脈が失われる。毎回ゼロから関係構築 |

### Broker

| ニーズ | 現状の解消方法 | 犠牲にしていること |
| ------ | -------------- | ------------------ |
| 自分の価値を守る | 情報をあえて不透明に保ち「自分がいないと困る」状態を維持 | 業界全体の非効率を温存する |
| 単価の確保 | 情報の非対称性を交渉の武器にする | 信頼関係の構築より情報優位に依存する |

### Vendor

| ニーズ | 現状の解消方法 | 犠牲にしていること |
| ------ | -------------- | ------------------ |
| SE稼働率の安定 | 仲介会社と関係を作り「優先案件」を依頼 | 仲介へのマージンを払い続ける |
| SE引き抜き防止 | 契約書の条項・待遇改善 | 根本解決にならない。SEの意思は止められない |
| 自社SEの実績管理 | Excelや紙のスキルシートで管理 | 更新が属人的。退職でデータが消える |

### 個人SE

| ニーズ | 現状の解消方法 | 犠牲にしていること |
| ------ | -------------- | ------------------ |
| 市場価値の把握 | 転職エージェントに聞く・同業の口コミ | 客観的なデータではない |
| 実績の証明 | 職務経歴書・自作ポートフォリオ | 第三者による証明がない。転職で更新が必要 |
| 単価向上 | 仲介会社に交渉・転職繰り返し | 交渉材料が乏しく、構造的な改善は難しい |

### Team Integrator

| ニーズ | 現状の解消方法 | 犠牲にしていること |
| ------ | -------------- | ------------------ |
| 良いBP調達 | 個人の人脈リストに連絡・紹介依頼 | 人脈の広さに上限がある。品質にばらつき |
| 体制の実績証明 | 口コミ・紹介のみ | スケールしない。数値化できない |

## 6. プラットフォームが代替する機能

上記の「現状の解消方法」の限界を、プラットフォームが構造的に置き換える。

### Client向け機能

| 解消したい限界 | プラットフォーム機能 |
| -------------- | -------------------- |
| 一斉メール・量でカバー | 条件指定による絞り込み検索。トラストスコアによる品質保証 |
| 面談頼みのスキル確認 | 就業実績の第三者証明。過去Client評価の閲覧 |
| 書類だけのコンプライアンス | 多重商流の自動検出・アラート |
| 担当者変更時のリセット | 企業間取引メモリ（担当者が変わっても文脈が残る） |

### Broker向け機能

| 解消したい限界 | プラットフォーム機能 |
| -------------- | -------------------- |
| 情報優位だけの価値 | コーディネーター実績スコア（マッチングの定着率・条件伝達の正確さ）。情報だけでなく調整能力で評価される |
| 単純仲介の代替不安 | 複雑案件・特殊スキル案件のコーディネーター専門枠。高付加価値Broker認定 |

### Vendor向け機能

| 解消したい限界 | プラットフォーム機能 |
| -------------- | -------------------- |
| 仲介依存でしか案件が来ない | 要員会社が案件に直接エントリーできる仕組み |
| SE稼働率の不安定 | 先物マッチング（要員会社が稼働終了予定を登録し、次案件を先手で確保） |
| SE引き抜きの恐怖 | 要員会社経由の先物設計（SEへのオファーは要員会社が受け取り、SEに提示する） |
| 実績・スキルの属人管理 | SEスキル・稼働実績のプラットフォーム管理（担当者が変わっても消えない） |

### 個人SE向け機能

| 解消したい限界 | プラットフォーム機能 |
| -------------- | -------------------- |
| 市場価値が分からない | 市場単価インデックス（同スキル・同経験年数の相場） |
| 実績が消える | 就業実績の個人アカウントへの蓄積（会社を変えても持ち運べる） |
| 単価交渉の根拠がない | 自分への「N社がオファー中」の通知。市場での需要が見える |

### Team Integrator向け機能

| 解消したい限界 | プラットフォーム機能 |
| -------------- | -------------------- |
| 人脈依存のBP調達 | スコア・実績で絞り込めるBP検索 |
| 体制実績の数値化不可 | TIスコア（体制完走率・BP定着率・Client NPS） |
| マージン構造の不透明さ | 当事者のみ開示設計（BPは自分の受取単価が分かる。Clientは体制単価合計のみ） |

## 7. プラットフォームのゴール

### 各プレイヤーにとってのゴール

| プレイヤー | プラットフォームが実現する状態 |
| ---------- | ------------------------------ |
| Client | 信頼できる相手を、根拠を持って、直接調達できる |
| Vendor | 自社SEの稼働を自分でコントロールでき、仲介依存から脱せる |
| Broker | 情報優位ではなくコーディネーション能力で選ばれる存在になれる |
| 個人SE | 自分の実績・市場価値が自分のものとして蓄積・証明される |
| Team Integrator | 人脈ではなく実績で体制を組み、その価値が客観的に見える |

### 業界全体のゴール

> **取引の信頼を、人脈ではなくデータが担保する構造に変える。**

現在の多重下請け構造が存在する根本原因は「信頼の証明手段が人脈しかない」ことだ。
人脈がないと信頼できる相手を見つけられないから、人脈を持つ仲介会社が必要になる。
人脈に依存するから不透明な構造が温存される。

プラットフォームは「取引の実績・評価のデータ」という新しい信頼の証明手段を作ることで、
この構造を根本から変える。

信頼をデータが担保できれば：

- 直接取引ができる（仲介を挟む必要がない）
- 多重商流が自然に短縮される
- SEの実績が人脈を超えてポータブルになる
- 仲介会社は「人脈の独占」ではなく「コーディネーション能力」で勝負できる

### 「戻れなくなる」状態の定義

プラットフォームが成功した状態とは、各プレイヤーがこう感じることだ：

- Client：「プラットフォームの取引履歴と評価なしに、見知らぬ会社と取引することが怖くなった」
- Vendor：「自社SEの稼働計画をプラットフォーム外で管理することが非効率に見えるようになった」
- 個人SE：「自分の実績がここに蓄積されているので、これを捨てて別のやり方に戻れない」
- Team Integrator：「TIスコアがある状態で次の体制オファーを取りに行くのと、ない状態で取りに行くのでは結果が違いすぎる」
